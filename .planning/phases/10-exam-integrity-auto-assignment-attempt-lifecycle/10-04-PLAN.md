<objective>
Build the phase's one voiding authority, and close the last unguarded locked read on `attempts`.

Purpose: D-2 mandates **ONE** shared, lock-guarded voiding service — never duplicated across two
controllers — mirroring the `AttemptGrader` precedent. CLS-07 (plan 07) and EDT-04 (plan 08) both
converge on it. Its delete must take the **same row lock** `Attempt::lockAndFinalize()` takes, so a
racing student autosave hits the already-built `AttemptVanishedException` instead of a 500.

D-5 rides along here because D-2's hard delete is what promotes that race from exotic to routine: it
stops needing a student to delete their own account mid-grade and becomes "a lecturer saves a grade
while another lecturer resets that exam."

Output: `AttemptVoider` (green against plan 01's spec), the D-5 guard, and a lecturer-safe
`AttemptVanishedException` (green against plan 02's spec).
</objective>

**`public function summarize(Exam $exam): array`**

Compute all five UI-SPEC counts from ONE grouped aggregate query. `attempts.status` already IS the
graded/ungraded distinction — `AttemptGrader::syncStatus()` flips it to `graded` the moment every
open-text answer has a score — so the three buckets map 1:1 onto the three literal status values:

  - query: `Attempt::where('exam_id', $exam->id)->selectRaw('status, count(*) as aggregate')->groupBy('status')->pluck('aggregate', 'status')`
  - `$inProgress = (int) ($counts['in_progress'] ?? 0);`
  - `$submittedUngraded = (int) ($counts['submitted'] ?? 0);`
  - `$graded = (int) ($counts['graded'] ?? 0);`
  - return array keys exactly: `inProgress`, `submittedUngraded`, `graded`,
    `notYetGraded` (= inProgress + submittedUngraded), `total` (= all three).

Cast every value with `(int)`. Plan 01's spec uses `assertSame`, and the counts are interpolated into
Blade copy downstream — the int/string distinction is load-bearing.

**Do NOT add a `whereHas('answers', ...)` completeness check.** The UI-SPEC's prose describes
`$submittedUngraded`/`$graded` as derived conditions ("status = submitted AND not fully graded"); that
description is imprecise. Deriving it would duplicate work `AttemptGrader::syncStatus()` already does
at grade-save time and could disagree with it — two sources of truth for the number that decides
whether a lecturer destroys graded work. Add a short doc comment on `summarize()` saying exactly this,
so nobody "fixes" it back toward the UI-SPEC's wording later.

**`public function void(Exam $exam): int`**

Wrap in `DB::transaction()`. Inside:
  - `$ids = Attempt::where('exam_id', $exam->id)->lockForUpdate()->pluck('id');`
  - `if ($ids->isEmpty()) { return 0; }`
  - `Attempt::whereIn('id', $ids)->delete();`
  - `return $ids->count();`

Takes an `Exam` model, never a raw ID and never request input — matching
`AttemptGrader::handleFinalized(Attempt $attempt)`'s convention and the codebase's CWE-915 discipline
(see `AnswerGradeController.php:31`'s "Explicit single-key write — never `$request->all()`" comment).

Doc-comment the lock, because the reason is non-obvious: InnoDB's `DELETE` does take implicit row
locks, so the explicit `SELECT ... FOR UPDATE` first is not what makes the delete safe — it is what
makes the **serialization point** unambiguous and matches `Attempt::lockAndFinalize()`'s established
idiom. Whichever transaction's `FOR UPDATE` read commits first wins; the loser blocks, re-reads, finds
its row gone, and throws `AttemptVanishedException` (Phase 9, already built and tested). That is D-2's
"same row lock" requirement.

Also doc-comment why `void()` needs no null-guard of its own: it operates on a **set** (`whereIn`), so
zero rows is a legitimate empty-set outcome (`isEmpty()` → 0), not a vanished-row error. The exception
fires on the other side of the race, in code that already exists. Do not reimplement it here.

Answers need no manual cleanup — `answers.attempt_id` cascades at the FK layer.
