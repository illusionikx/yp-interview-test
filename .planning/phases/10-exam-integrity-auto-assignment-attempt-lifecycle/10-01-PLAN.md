**THE FACTORY TRAP — read this before writing a line.** `ExamFactory` and `SectionFactory` EACH
independently call `Subject::factory()`. So a bare `Exam::factory()->create()` plus a bare
`Section::factory()->create()` already land in different subjects **by accident**. That means this
entire negative test could pass while proving NOTHING — green, and guarding an open CRITICAL leak.
Never let factory defaults decide the thing under test.

Write two test methods:

**Method 1 — `test_a_student_enrolled_only_in_a_different_subject_cannot_see_open_or_start_the_exam()`**
(INT-04, threat T-10-01). Build the different-subject case DELIBERATELY:
  - `$subjectA = Subject::factory()->create();` and `$subjectB = Subject::factory()->create();`
  - `$this->assertNotSame($subjectA->id, $subjectB->id, 'Fixture is not exercising the cross-subject case — the negative assertions below would pass vacuously.');`
    This guard assertion is MANDATORY and is the whole point of the method. Without it, a future
    factory change that made both subjects identical would silently turn this into a no-op.
  - `$examOnA = Exam::factory()->published()->for($subjectA)->create();`
  - `$sectionOnB = Section::factory()->for($subjectB)->create();` — use `->for()`, never a bare
    `Section::factory()->create()`, so the section's subject is pinned, not random.
  - `$student = User::factory()->student()->create();`
  - `$sectionOnB->enrollments()->attach($student->id, ['status' => EnrollmentStatus::Enrolled]);`

  Then assert all THREE surfaces independently — per D-1 and the v2.0 lesson that a list and its gate
  must never diverge. A leak that is filtered out of the list but reachable by URL is still the leak:
  1. LIST: `$this->assertFalse(Exam::visibleTo($student)->whereKey($examOnA->id)->exists());`
  2. DIRECT ACCESS: `actingAs($student)->get(route('student.exams.show', $examOnA))` → `assertForbidden()`
  3. START (the write path): `actingAs($student)->post(route('student.attempts.store', $examOnA))` → `assertForbidden()`
     — INT-04 says "see **or start**"; the existing regression test only covered list+takeable, never
     the store() write path. Also assert no row landed:
     `$this->assertDatabaseMissing('attempts', ['exam_id' => $examOnA->id, 'user_id' => $student->id]);`

**Method 2 — `test_a_student_enrolled_in_the_exams_own_subject_can_see_open_and_start_it()`**
(CLS-05's positive counterpart — the control that proves method 1 fails for the RIGHT reason).
This method is what stops method 1 from being satisfiable by a `scopeVisibleTo()` that simply denies
everyone. Pin both to the SAME subject explicitly:
  - `$subject = Subject::factory()->create();`
  - `$exam = Exam::factory()->published()->for($subject)->create();`
  - `$section = Section::factory()->for($subject)->create();`
  - `$this->assertSame($exam->subject_id, $section->subject_id, 'Fixture must pin exam and section to the same subject — see the factory trap.');`
  - enrolled student as above, then assert the mirror-image of method 1: visible in list (`assertTrue`),
    `assertOk()` on show, and `assertRedirect()` (not forbidden) on attempts.store.

Both methods need a class-level doc comment stating: this file is INT-04's acceptance; the
`assertNotSame`/`assertSame` subject-ID guards are load-bearing, not decoration; and the pair
(negative + positive) must always be maintained together.

Create `Tests\Feature\AttemptVoiderTest` using `RefreshDatabase`.

This is the phase's single most safety-critical test. D-2 hard-deletes attempts with no undo and no
audit trail, so the warning is the ONLY thing between a lecturer and permanent destruction of graded
student work. A modal that reports "0 graded" while 3 are graded is the catastrophic failure this
test exists to prevent. Assert the exact numbers — never merely that a modal appears.

**Note the count semantics, which are simpler than the UI-SPEC's prose implies.** The UI-SPEC
describes `$submittedUngraded` and `$graded` as derived conditions ("status = submitted AND not fully
graded"). They are not. `attempts.status` is a plain three-value string column
(`in_progress` / `submitted` / `graded`) that `AttemptGrader::syncStatus()` already flips to `graded`
the moment every open-text answer has a score. So the three buckets map 1:1 onto the three literal
status values. Build the fixture from the factory states directly; do not add any per-answer
completeness check.

**Method 1 — `test_summarize_reports_all_five_counts_exactly()`** (INT-02, threat T-10-02).
Fixture — exactly one attempt in each of the three states, each for a distinct student so the
`attempts.unique(exam_id, user_id)` index is not violated:
  - `$exam = Exam::factory()->published()->create();`
  - one `Attempt::factory()->for($exam)->for(User::factory()->student())->create();` (default state is `in_progress` — do not override it)
  - one `...->submitted()->create();`
  - one `...->graded(5)->create();`

Act: `$counts = app(AttemptVoider::class)->summarize($exam);`

Assert all five keys with `assertSame` (not `assertEquals` — the values are ints and the string/int
distinction matters for the Blade interpolation downstream):
  - `inProgress` === 1
  - `submittedUngraded` === 1
  - `graded` === 1
  - `notYetGraded` === 2  (inProgress + submittedUngraded)
  - `total` === 3

**Method 2 — `test_summarize_reports_zeroes_for_an_exam_with_no_attempts()`**. All five keys === 0.
This pins the zero case the UI-SPEC's disabled-trigger / no-modal branch depends on.

**Method 3 — `test_summarize_counts_only_the_given_exams_attempts()`**. Create a SECOND exam with its
own attempts in all three states, then assert `summarize($exam)` is unaffected. Without this, a
`summarize()` that forgot its `where('exam_id', ...)` clause would still pass method 1 — and would
report another exam's graded work as being at risk on this exam's modal.

**Method 4 — `test_void_hard_deletes_every_attempt_and_cascades_to_answers()`**. Build an exam with a
graded attempt that HAS answer rows (use `Answer::factory()` against a question of that exam). Assert
`void($exam)` returns the count deleted, that `assertDatabaseMissing('attempts', ...)` holds, and —
critically — that `assertDatabaseMissing('answers', ['attempt_id' => $attempt->id])` holds too. This
pins D-2's stated consequence: the cascade destroys graded scores permanently. If a future change
made `answers.attempt_id` restrict instead of cascade, this test must fail loudly.

**Method 5 — `test_void_on_an_exam_with_no_attempts_returns_zero_and_throws_nothing()`**. The
empty-set case is a legitimate outcome, not an error.

Class doc comment must state: this file is INT-02's acceptance; count-correctness is a security
control (T-10-02), not a UX detail, because D-2's delete is irreversible.
