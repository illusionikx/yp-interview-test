# Phase 10 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope of this contract:** exactly the small UI surface this phase actually ships — the CLS-06
draft/active toggle, the CLS-07 reset-submissions warning, the EDT-04 save-an-attempted-exam warning,
and the outcome toasts for all of the above (success criterion 5). All of it lives on the **existing**
`lecturer/exams/show.blade.php`, `lecturer/exams/edit.blade.php`, and the two question forms — because
Phase 12 has not built the two-tab workspace yet. Phase 12 will *relocate* these controls into its
exams tab; it must reuse the service and the copy this contract defines, not re-derive them.

**Explicitly out of scope (do not design here):** the two-tab lecturer workspace shell (Phase 12), the
student-facing class/exam list (Phase 13), the dashboard (Phase 11). This phase touches only the
lecturer exam-detail surfaces that already exist.

**No new component.** This contract specifies how Phase 9's `<x-confirm-modal>` (blocking) and
`<x-toast>` (non-blocking) are invoked for this phase's destructive actions — it does not add a third
component, per the researcher brief's explicit instruction.

## Page & Component Contracts

### 1. Exam show page (`resources/views/lecturer/exams/show.blade.php`)

**A. Status toggle (CLS-06) — reuses the existing Publish/Unpublish buttons, behavior changes only:**

- Keep the existing copy exactly as shipped: pill reads "Published" / "Draft"
  (`<x-status-pill :status="$exam->is_published ? 'published' : 'draft'">`, unchanged — no relabel to
  "Active", per the minimal-risk-copy decision below), button reads "Publish" (draft state) /
  "Unpublish" (published state) — **do not rename these to "Activate"/"Deactivate."** CLS-06's
  requirement is about the toggle's *bidirectionality*, not its label; renaming shipped, tested copy
  for no functional gain violates the project's "minimal, correct implementation" constraint. This is
  Claude's discretion, exercised in favor of the smaller diff.
- **Behavior change (the actual CLS-06 work):** `ExamController::unpublish()` currently blocks with
  `if ($exam->attempts()->exists()) { return back()->with('status', 'Cannot unpublish...'); }`. This
  block must be **removed** — CLS-06 explicitly requires the toggle to work "in both directions,
  including after students have already attempted it." Toggling does **not** delete or touch any
  attempt row; it only flips `is_published`. No confirm-modal is needed for the toggle itself — it is
  non-destructive (existing attempts are untouched either direction), so it stays the single-click
  action it already is.
- **Toast on toggle (success criterion 5):**
  - Publish: `session('status', 'Exam published. Students can now see and start it.')`
  - Unpublish: `session('status', 'Exam moved back to draft. Students can no longer start it, but existing attempts are unaffected.')` — the second clause matters: it tells the lecturer explicitly that toggling is safe and does not touch attempt data, which is the one fact a lecturer needs before using this control freely.

**B. Edit becomes reachable regardless of publish state (D-4/EDT-04) — the `@unless ($exam->is_published)` wrapper around the "Edit" link is removed:**

- The "Edit" link and the "Add a question" panel (currently hidden once published) become **always**
  visible. Per-question "Edit"/"Delete" links (currently `@unless ($exam->is_published)`-gated) become
  always visible too. This is the direct, mandatory consequence of D-4 relaxing all three Form
  Requests (`UpdateExamRequest`, `StoreQuestionRequest`, `UpdateQuestionRequest`).
  - **`abort_if($exam->is_published, 403)` inside `ExamController::destroy()` (exam deletion) is
    UNCHANGED — deleting the whole exam stays draft-only.** Only *editing* (exam details, questions,
    options) relaxes. Do not conflate the two gates.
- Editing an exam/question with **zero** existing attempts behaves exactly as it does for a draft exam
  today — a plain save, no modal, `session('status', 'Exam updated.')` (or the existing
  question-store/update flash, unchanged).
- Editing an exam/question with **one or more** existing attempts is where EDT-04's warning fires —
  see the shared "Destructive warning" contract below.

**C. "Assign to sections" panel is DELETED entirely (D-1 / FIX-03-resolved-by-removal):**

- The whole `<div>` block containing the "Assign to sections" heading, the section checkboxes, and the
  "Update assignment" submit button is removed from this view — the `exam_section` pivot it wrote to no
  longer exists. `ExamAssignmentController` and its route (`lecturer.exams.assignment.update`) are
  deleted along with it (backend work, noted here only because it changes this page's layout).
  **This is the FIX-03 resolution:** the bug ("Update assignment gives no visible feedback") cannot
  exist once the button that caused it is gone. Do not build a toaster for a screen that no longer
  exists.

**D. New "Submissions" panel (CLS-07) — replaces the deleted "Assign to sections" panel's position in the page's vertical flow:**

```
┌─────────────────────────────────────────────────────────┐
│  Submissions                                    (h3, 20px/600)
│                                                            │
│  12 students have started this exam but have not been    │  ← body, 16px/400, text-body
│  graded, and 4 have already been graded.                 │
│                                                            │
│  [ Reset submissions ]  ← outline red trigger button      │
└─────────────────────────────────────────────────────────┘
```

- Heading: "Submissions" (matches the existing "Questions" panel's `h3.text-xl.font-semibold` style
  exactly — same card, same padding).
- **Summary line above the button** (always visible, not inside the modal) — gives the lecturer the
  stakes *before* they click, not just at confirm time:
  - Zero attempts of any kind: *"No students have started this exam yet."* — muted `text-body`, and
    the "Reset submissions" button renders **disabled** (`opacity-50 cursor-not-allowed`, no `href`/
    `@submit.prevent` wiring) rather than hidden — a lecturer should see the control exists and
    understand why it's inert, not wonder if the feature is missing.
  - In-progress/ungraded only (graded = 0): *"{N} student(s) have started this exam but have not been
    graded."*
  - Graded > 0: *"{N} student(s) have started this exam but have not been graded, and {M} have already
    been graded."* — the `{M} have already been graded` clause renders in the destructive-red text
    color (`text-red-700 dark:text-red-400`) as the one piece of accent-red text on this page outside
    the modal/button, because it is the single fact most likely to change a lecturer's mind.
- **Trigger button — "Reset submissions" — style is deliberately NOT the modal's filled-red danger
  style.** It is a muted red *outline* button (`border border-red-300 text-red-700 hover:bg-red-50
  dark:border-red-800 dark:text-red-400 dark:hover:bg-red-950/30`), same size/padding as the other
  action buttons on this page. Rationale: the filled-red "delete" treatment is reserved for the
  modal's actual confirm action (where the commitment happens); the page-level trigger that merely
  *opens* the conversation should read as "proceed with caution," not "you are about to delete right
  now" — that distinction is what stops accidental double-intensity red fatigue across the page.
- Disabled state (zero attempts) needs no modal at all — see "Design Decisions" below.

### 2. Exam edit page (`resources/views/lecturer/exams/edit.blade.php`) and question forms (`.../questions/_form.blade.php`, `.../questions/edit.blade.php`)

**EDT-04's warning fires here, on save, only when `$exam->attempts()->exists()`.** All three views
(exam-details edit, question create, question update) share the identical interception pattern:

- The controller (`ExamController::edit()`, and the two question-form-rendering actions) passes
  `$attemptCounts` (or equivalent — shape is the planner's call, see "Shared summary source" below) to
  the view.
- If `$attemptCounts['total'] === 0`: the "Save changes" button is a plain `type="submit"` — **no
  interception, no modal.** This is the answer to "what does the modal say at zero counts" — it isn't
  shown at all; friction is proportional to actual risk.
- If `$attemptCounts['total'] > 0`: the form gets `x-data` + `@submit.prevent="$dispatch('open-modal',
  'save-exam-changes')"` (mirroring the existing delete-form interception pattern in
  `show.blade.php:49`), the submit button's `type="submit"` becomes effectively gated behind the modal,
  and `<x-confirm-modal name="save-exam-changes" ... x-on:click="$refs.editExamForm.submit()" />` is
  rendered once on the page. See the "Bridging copy" contract below for exact title/body — **this is
  the part of the contract that matters most**, because the lecturer's mental model walking in is "I'm
  saving a form," not "I'm about to delete data," and the copy must make that translation explicit
  rather than assume it.

### Shared "destructive warning" content contract (CLS-07 + EDT-04)

Both warnings are driven by the **same three counts**, computed from the same source (the voiding
service D-2 mandates, e.g. `AttemptVoider::summarize(Exam $exam): array` mirroring `AttemptGrader`'s
shape — exact method name is the planner's call, but **compute it once, in one place, and have both
the show page and the edit/question pages call it** — do not let the "Submissions" panel's summary line
and the EDT-04 modal's body drift out of sync by being computed twice):

| Count | Definition |
|-------|-----------|
| `$inProgress` | `attempts` where `status = 'in_progress'` for this exam |
| `$submittedUngraded` | `attempts` where `status = 'submitted'` AND not fully graded (some/all gradable answers still lack `graded_at`) |
| `$graded` | `attempts` where `status = 'submitted'` AND fully graded |
| `$notYetGraded` | `$inProgress + $submittedUngraded` — the single number used in every "have not been graded" sentence below |
| `$total` | `$inProgress + $submittedUngraded + $graded` — the button label's count and the service's actual delete count |

**Why `$notYetGraded` folds two states into one number:** the roadmap names exactly two populations
("started but not finished" vs. "graded"). A third state — submitted but still awaiting grading — is
real (an essay answer sitting ungraded) and would otherwise be silently uncounted by either named
bucket. Folding it into the "not yet graded" line keeps the contract's language honest (nothing is
omitted) without inventing a third line the roadmap didn't ask for. If a planner prefers a literal
three-line breakdown instead, that is acceptable — but the two-number version above is the
recommended default because it satisfies INT-02's "concrete count" requirement with less surface area
to keep synced across two forms and one summary line.

**CLS-07 modal (`reset-submissions`):**

| Field | Value |
|-------|-------|
| Title | "Reset exam submissions?" |
| Body ($graded = 0) | "{$notYetGraded} student(s) have started this exam but have not been graded. Resetting will permanently delete {$total} attempt(s) so they can start again. This cannot be undone." |
| Body ($graded > 0) | "{$notYetGraded} student(s) have started this exam but have not been graded, and {$graded} student(s) have already been graded. Resetting will permanently delete all {$total} attempts — including the {$graded} graded score(s). This cannot be undone." |
| Confirm label | "Reset {$total} submissions" |
| Cancel label | "Cancel" (default) |
| `danger` | `true` |

**EDT-04 modal (`save-exam-changes`), rendered on the exam-edit page and both question forms:**

| Field | Value |
|-------|-------|
| Title | "Save changes and reset attempts?" |
| Body ($graded = 0) | "{$notYetGraded} student(s) have started this exam but have not been graded. Saving your changes will cancel their attempt(s) so they can start over. This cannot be undone." |
| Body ($graded > 0) | "{$notYetGraded} student(s) have started this exam but have not been graded, and {$graded} student(s) have already been graded. Saving your changes will permanently delete all {$total} attempts — including the {$graded} graded score(s). This cannot be undone." |
| Confirm label | "Save & reset {$total} attempt(s)" |
| Cancel label | "Cancel" — clicking Cancel returns to the form with **nothing submitted and nothing changed** (closes the modal only; the lecturer's typed edits remain in the still-open form so they don't lose their in-progress edit by backing out of the destructive part) |
| `danger` | `true` |

**Toast on completion:**

- CLS-07 reset, successful: `session('status', "Reset {$total} submission(s) for \"{$exam->title}\".")`
- EDT-04 save that triggered voiding: `session('status', "Exam updated. {$total} affected attempt(s) were reset.")` — deliberately **not** the plain "Exam updated." used for a no-attempts save; the lecturer must see the side effect happened, not just that the save succeeded. This is success criterion 5's second concrete instance (the first is the publish/unpublish toast above).
- EDT-04 save with zero attempts (no voiding triggered): unchanged existing `session('status', 'Exam updated.')`.

### Design Decisions (answering the brief's open questions explicitly)

1. **Zero-count behavior:** no modal is ever shown at zero. The CLS-07 trigger renders disabled with a
   muted explanatory line instead of a button; the EDT-04 save path skips interception entirely and
   submits directly. Friction only exists where risk exists.
2. **In-progress-only (graded = 0) vs. graded > 0:** both use the identical modal shape (`danger=true`,
   same title) — the only difference is the second sentence appearing (or not) and the `text-red-700`
   emphasis on the graded clause in the page-level summary line. Two modal *shapes* (severity tiers)
   would add complexity the roadmap doesn't call for; one shape with conditional sentence content
   already differentiates the populations as required.
3. **Confirm button naming the act:** **yes.** Both modals use "Reset {N} submissions" / "Save & reset
   {N} attempt(s)" rather than a generic "OK"/"Confirm"/"Delete" — the count and the verb together are
   the load-bearing part of the copy, consistent with the existing 3 call sites' pattern of naming the
   object being deleted ("Delete exam?" not "Are you sure?").
4. **Type-to-confirm:** **not recommended.** The existing `<x-confirm-modal>` component has no
   type-to-confirm affordance and adding one means extending the shared component with new state
   (a bound text input + a disabled-until-match confirm button) that every other call site (3 existing
   + this phase's 2 new ones) would need to either adopt or explicitly opt out of. That is exactly the
   kind of "feature breadth over minimal correctness" CLAUDE.md's scope-discipline constraint rejects.
   The concrete-count summary line (visible *before* the click) + the named-count confirm button
   (visible *at* the click) is judged sufficient friction for a project-scale, single-lecturer-session
   destructive action — it is the same friction level already accepted for the three existing delete
   confirmations, which this phase's two new ones are extending, not escalating past.
5. **EDT-04's "I'm saving" → "this deletes data" bridge:** the modal title itself does the bridging —
   "Save changes **and reset attempts**?" — naming both halves of the action in one sentence, rather
   than a generic "Are you sure?" that would leave the lecturer to infer the connection. The body's
   first clause states the population at risk before the second clause states the consequence
   ("Saving your changes will cancel..."), so the causal link between clicking Save and data loss is
   explicit, not implied.
