mutation warn-then-void atomically.

Purpose: the mechanism protecting graded scores moves from "you may not edit an attempted exam at all"
to "editing destroys its attempts, and you are told first". That is strictly more permissive but
explicitly warned — and it is what CLS-06's toggle relaxation (plan 05) already depends on.

reading. Relaxing only three ships a half-EDT-04: a lecturer could edit an attempted exam's text but
not remove a question, for no principled reason, and the "always visible" Delete link would 403 on

**The flow, identical at all four mutations:**

1. `$counts = app(AttemptVoider::class)->summarize($exam);` — **BEFORE** the write. Used to decide
   whether voiding applies and to build the toast.
2. The write (`update()` / `create()` / `delete()`), then `void($exam)` **only if `$counts['total'] > 0`**
   — both inside **ONE** `DB::transaction()` (D-7).
3. Redirect with the outcome flash.

**Ordering is a safety property, not a style choice.** Void must come after a successful write, never
before. Validation runs in the Form Request, before the controller method is entered, so a validation
failure never reaches the transaction at all — which is exactly what
`test_a_failed_validation_on_an_attempted_exam_destroys_no_attempts` pins. A void that ran first would
destroy every student's work for a save that never landed: the worst possible outcome.

**D-7 — one transaction, both or neither.** Research recommended two sequential transactions
(save-then-void); the orchestrator overrode that. With two, a save that succeeds while the void fails
leaves an edited exam with stale attempts still attached — precisely the inconsistency EDT-04 exists to
prevent. The reverse order is worse. Only an atomic pair gives both-or-neither, and correctness on a
permanently destructive path outranks lock-hold time at this project's scale.

Note that `AttemptVoider::void()` opens its own `DB::transaction()`. Calling it inside your outer
transaction is correct and intended — Laravel implements the nested call as a savepoint, so the outer
transaction stays the atomic unit. Do not refactor `void()` to remove its transaction; it is also called
standalone by `resetSubmissions()` (plan 07).
