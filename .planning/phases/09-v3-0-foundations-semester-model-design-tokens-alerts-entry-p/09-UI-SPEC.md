# Phase 9 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope of this contract:** exactly four UI surfaces — (1) the landing page, (2) the Flowbite login
card, (3) the `<x-toast>` component, (4) the `<x-confirm-modal>` component. SEM-01..03 and INT-01 are
backend-only with **one** copywriting exception (the attempt-vanished error toast, below). No
dashboard, subject list, or navigation-restructure surface belongs in this file — those are Phase 11.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding, gap between stacked toasts |
| sm | 8px | Compact element spacing, toast icon-to-text gap |
| md | 16px | Default element spacing, landing top-bar horizontal padding |
| lg | 24px | Section padding, modal body padding, login card padding (`p-6`) |
| xl | 32px | Layout gaps, hero vertical rhythm |
| 2xl | 48px | Major section breaks (top bar to hero on landing) |
| 3xl | 64px | Page-level spacing (landing page top/bottom breathing room) |

Exceptions: **the login card is reproduced verbatim from `v3.md` lines 3–26** and its exact Flowbite 4
classes are NOT adjusted to the 8-point scale — `py-2.5` (10px) and `mb-2.5` (10px) ship as supplied.
Do not "fix" these to `py-2` or `py-3` — pixel-exact reproduction of the supplied design is the
acceptance bar for NAV-02, not scale purity. All other new markup in this phase (landing hero, top
bar, toast, confirm modal) follows the 8-point scale with no further exceptions.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `bg-neutral-primary-soft` (white / `gray-800` dark) | Page background, card surfaces, modal surface |
| Secondary (30%) | `bg-neutral-secondary-medium` (`gray-50` / `gray-700` dark) | Input fields, top bar background, toast container background |

Accent reserved for: **the login submit button, the landing hero's "Sign in" CTA, all focus rings,
the checked checkbox state, and the two brand-colored inline links** — nowhere else. Every other
interactive element (secondary/cancel buttons, nav items, toast close buttons) stays neutral gray.
Destructive red is reserved for the confirm-modal's confirm button and error-toast accent only —
never for a general warning or informational state.

## Design Token Port (UI-03) — concrete values

This is the actual blocker fix. Add to `tailwind.config.js` → `theme.extend`, and define the CSS
custom properties it references in `resources/css/app.css` (flipped by the existing `.dark` class,
per Decision #5/the CONTEXT.md dark-mode mechanism).

### `tailwind.config.js` additions

```js
theme: {
  extend: {
    fontFamily: { sans: ['Figtree', ...defaultTheme.fontFamily.sans] }, // unchanged
    colors: {
      brand: {
        DEFAULT: 'rgb(var(--color-brand) / <alpha-value>)',
        strong:  'rgb(var(--color-brand-strong) / <alpha-value>)',
        medium:  'rgb(var(--color-brand-medium) / <alpha-value>)',
        soft:    'rgb(var(--color-brand-soft) / <alpha-value>)',
      },
      'fg-brand': 'rgb(var(--color-fg-brand) / <alpha-value>)',
      heading:    'rgb(var(--color-heading) / <alpha-value>)',
      body:       'rgb(var(--color-body) / <alpha-value>)',
      default:            'rgb(var(--color-border-default) / <alpha-value>)',
      'default-medium':   'rgb(var(--color-border-default-medium) / <alpha-value>)',
      neutral: {
        'primary-soft':     'rgb(var(--color-neutral-primary-soft) / <alpha-value>)',
        'secondary-medium': 'rgb(var(--color-neutral-secondary-medium) / <alpha-value>)',
      },
    },
    borderRadius: { base: '0.5rem' },   // → rounded-base
    boxShadow:   { xs: '0 1px 2px 0 rgb(0 0 0 / 0.05)' }, // → shadow-xs
  },
},
```

Why this covers every class the login snippet uses without a separate `borderColor`/`ringColor`
block: Tailwind 3's stock preset defines `borderColor` and `ringColor` as *functions of* `theme('colors')`,
so extending `colors` alone makes `border-default`, `border-default-medium`, `ring-brand`,
`ring-brand-medium`, `ring-brand-soft`, `text-fg-brand`, `bg-brand-strong`, and
`placeholder:text-body` all resolve — no duplicate color definitions needed.

- **Hero** (single section, vertically centered, no features/how-it-works, no role-picker):
  - Display: "Online Examination Portal" (36px/600/1.2)
  - Subtitle directly beneath: "for Yayasan Peneraju Technical Assessment" (Heading role, 20px/600/1.2,

  - One-line description (Body role, 16px/400/1.5): e.g. *"Take timed exams, view results, and manage
    assessments — all in one place."* (exact wording is Claude's discretion; must stay one line)
  - Primary CTA: **"Sign in"** button, `bg-brand` filled, links to `route('login')` — the phase's

- Background: `bg-neutral-primary-soft`; no card/box around the hero — full-bleed page.

### Login card (`resources/views/auth/login.blade.php`, NAV-02)

- Reproduce the `v3.md` lines 3–26 markup **structurally and class-for-class**: outer card
  (`w-full max-w-sm bg-neutral-primary-soft p-6 border border-default rounded-base shadow-xs`),
  `<h5>` heading "Sign in to our platform", email field, password field, remember-me checkbox +
  "Lost Password?" link on one row, full-width submit button "Login to your account", footer line
  "Not registered? Create account".
- **Deviations from the raw snippet required by NAV-02 and CONTEXT.md** (the raw snippet is a static
  Flowbite example with inert `#` hrefs — this app is not):
  - `<form action="#">` → `<form method="POST" action="{{ route('login') }}">` with `@csrf`.
  - Email/password inputs get `name`, `id` kept, `:value="old('email')"`, `required`,
    `autocomplete="username"` / `"current-password"`, and each is followed by
    `<x-input-error :messages="$errors->get('email')" />` (existing pattern) — validation errors stay
    inline per-field, they are NOT toasts (UX-03 governs create/save/delete, not form validation).
  - Remember-me checkbox gets `name="remember"`.
  - "Lost Password?" `href="#"` → `href="{{ route('password.request') }}"` (only if
    `Route::has('password.request')`, matching the existing Breeze guard pattern).
  - "Create account" `href="#"` → `href="{{ route('register') }}"`.
  - Keep `<x-auth-session-status>` (or fold its behavior into `<x-toast>` — see Sentinel Handling
    below) for the password-reset-redirect confirmation message.
- This page keeps using `layouts.guest` (or a trimmed variant) — it is still the pre-authentication
  shell; `<x-toast>` must be included here too (see below), not just in `layouts.app`.

### `<x-toast>` component (UX-02, UX-03)

**Placement:** fixed, top-right, below the navbar/top-bar (`top-20 right-4` or equivalent — must clear
whatever bar sits above it on both `layouts.app` and the landing/login guest shell), stacked with
`gap-2` (8px) between toasts, `z-50`.

**Trigger — zero controller changes:** reads `session('status')` and `session('error')` on page load
(the app's actual existing convention — see note below) and renders one toast per non-empty flash key
found. Include exactly once, in both `layouts/app.blade.php` and the guest/login shell, e.g.
`<x-toast />` right before `</body>` or immediately after the top bar.

**⚠️ Correction to CONTEXT.md's stated convention:** CONTEXT.md describes the flash keys as
`session('success')` / `session('error')`. The actual codebase (verified by grep across
`app/Http/Controllers`) flashes **`session('status')`** for every success-style message (35+ call
sites: "Exam created.", "Section deleted.", "You're enrolled in {section}.", etc.) and
**`session('error')`** for failure messages (enrollment/attempt error paths). `<x-toast>` must read
`status` + `error`, not `success` + `error` — there is no `session('success')` key anywhere in this
codebase today.

**⚠️ Sentinel exclusion list — do not toast these three literal `status` values:**
`verification-link-sent`, `password-updated`, `profile-updated`. These are Breeze-internal sentinel
strings that three existing views (`auth/verify-email.blade.php`,
`profile/partials/update-password-form.blade.php`,
`profile/partials/update-profile-information-form.blade.php`) already check for equality and render
their own dedicated inline confirmation text. If `<x-toast>` naively displays the raw session
`status` string whenever present, it will render the literal sentinel text ("password-updated") as a
toast bubble on the profile page, duplicating and garbling the existing inline confirmation. Skip the
toast render when `session('status')` exactly matches one of these three values; let those three
pages keep their existing behavior untouched.

**Variants and dismiss behavior:**
| Variant | Trigger | Auto-dismiss | Accent | Icon |
|---------|---------|---------------|--------|------|
| Success/info | `session('status')` (excluding sentinels above) | ~4000ms | neutral/green left border | check-circle |
| Error | `session('error')` | **never — persists until manually closed** | red left border | exclamation-circle |

Both variants always render a manual close button (×) regardless of auto-dismiss. **The × button is
icon-only and MUST carry `aria-label="Dismiss"`** — without it the control is unlabeled for screen
readers. Structure per
toast: icon (left) + message text (Body role, 16px/400) + close button (right). Container background
`bg-neutral-primary-soft`, `shadow-xs`, `rounded-base`, `border border-default` (reusing the ported
tokens — this is the second consumer of UI-03's token set beyond the login card).

**Copywriting — the attempt-vanished error (INT-01's only UI surface):** when
`Attempt::lockAndFinalize()`'s null-guard trips (the locked row vanished under a racing request), the
controller must redirect with an error flash reading: *"This exam attempt is no longer available.
Please return to your exam list."* — rendered through the same error-toast path, not a raw exception
page.

### `<x-confirm-modal>` component (UX-02)

**Purpose:** replaces all 3 existing `onsubmit="return confirm(...)"` call sites; blocking,
decision-required — distinct from the non-blocking toast.

**Call sites to migrate (verified locations):**
| File:line | Current copy | New modal title | New modal body |
|-----------|---------------|-----------------|-----------------|
| `lecturer/exams/show.blade.php:54` | "Delete this exam?" | "Delete exam?" | "This permanently removes “{{ $exam->name }}” and all its questions. This cannot be undone." |
| `lecturer/exams/show.blade.php:79` | "Delete this question?" | "Delete question?" | "This permanently removes this question and its options. This cannot be undone." |
| `lecturer/subjects/index.blade.php:38` | "Delete this subject?" | "Delete subject?" | "This permanently removes “{{ $subject->name }}”. Subjects with exams cannot be deleted." |

**Mechanism — must preserve existing form-submit semantics** (no fetch/AJAX, no controller changes):
each destructive form keeps its normal `method="POST"` submit target; remove `onsubmit="return
confirm(...)"` and instead intercept the submit with Alpine (`@submit.prevent` opens the modal;
confirming calls `$el.submit()` on the stored form reference). The three call sites become
`<x-confirm-modal :title="..." :body="..." danger>` wrapping (or paired via an Alpine store keyed by
`x-ref`) the existing `<form>` — implementation shape (wrapper vs. paired-by-id) is the executor's
call, but the observable behavior (a blocking dialog, Cancel + a red "Delete" button, no page
navigation until a choice is made) must match exactly.

**Props contract (for reuse by Phase 10's INT-02/CLS-07 warnings — do not hardcode these 3 messages
into the component itself):**
- `title` (string, required)
- `body` (string, required) — must be able to carry a dynamic count (Phase 10: "N students have
  started but not finished" vs. "N students have been graded — their scores will be permanently
  deleted.")
- `confirm-label` (string, default "Delete")
- `cancel-label` (string, default "Cancel")
- `danger` (bool, default true) — when true, confirm button uses the destructive red; Phase 10 may
  set this false for a non-destructive confirmation if one arises.

**Build on the existing `<x-modal>` primitive — do not write a second overlay/backdrop/focus-trap
from scratch.** `resources/views/components/modal.blade.php` already ships a complete, tested
primitive: `x-on:open-modal.window="$event.detail == '{name}' ? show = true : null"` /
`close-modal.window`, Escape-to-close, focus trap (Tab/Shift-Tab cycling), backdrop click-to-close,
and enter/leave transitions. `<x-confirm-modal>` should be a thin wrapper around `<x-modal :name="...">`
that adds the title/body/confirm/cancel slot content and dispatches
`window.dispatchEvent(new CustomEvent('open-modal', { detail: '{name}' }))` from each destructive
form's intercepted submit handler (Alpine `@submit.prevent`), storing a reference to the form being
confirmed (e.g. a small Alpine store or `x-ref` on the form) so the modal's "Delete" button can call
`.submit()` on the right form when clicked. This keeps the whole component consistent with the
codebase's one existing modal pattern instead of introducing a second, divergent one.

**Visual style:** the wrapped `<x-modal>` panel already uses `bg-white`/needs a `dark:bg-gray-800`
addition (existing primitive is light-mode-only today — verify and add dark classes as part of this
work, consistent with "existing views keep their `dark:` classes" scope, since this component itself
is new, not one of the ~28 legacy views Phase 14 owns) and `shadow-xl` — keep that elevation for the
modal panel; do not swap it for `shadow-xs` (that ported token is reserved for cards/inputs/toasts,
which sit flush against the page, not for a modal that must visually separate from the backdrop).
`rounded-base` may replace the primitive's `rounded-lg` on the panel for token consistency, or be left
as `rounded-lg` — cosmetically equivalent at this radius value (`0.5rem` both), executor's call.
