the now-duplicate `<x-auth-session-status>` call sites.

Create `resources/views/components/toast.blade.php`. It takes **no props** — per 09-CONTEXT.md's
"zero controllers" constraint it reads `session()` directly, so skip `@props()` entirely and open with
the `@php` block (this is the one documented deviation from the `status-pill.blade.php` shape).

In the `@php` block:
- Define `$sentinels = ['verification-link-sent', 'password-updated', 'profile-updated'];`
- `$status = session('status');`
- `$showStatus = $status && ! in_array($status, $sentinels, true);`
- `$error = session('error');`

Use a strict `in_array(..., true)` against that exact three-string list. Do NOT use a substring or
pattern match. Add a comment explaining the exclusion, because it is not guessable from this file:
these three are Breeze-internal sentinel values, not messages. Three shipped views test
`session('status')` for equality against them and render their own inline confirmation text —
`auth/verify-email.blade.php:6`, `profile/partials/update-password-form.blade.php:37`, and
`profile/partials/update-profile-information-form.blade.php:41,53`. Toasting them would render the
literal string "password-updated" as a bubble *next to* the existing "Saved." confirmation. Name those
three file paths in the comment so a future editor tempted to delete the list can see what breaks.

Also comment the key choice: this reads **`status`**, not `success`. `session('status')` is the app's
actual convention at 57 call sites; `session('success')` appears at zero. 09-CONTEXT.md's own text says
`success` and is wrong — the correction is recorded there and in 09-UI-SPEC.md. Note this explicitly so
nobody "fixes" the key back.

Wrap everything in `@if ($showStatus || $error)` so no empty container is emitted on the vast majority
of page loads.

Container: a single fixed-position div, top-right, clearing the bar above it on both shells, stacked
with an 8px gap, above other content in stacking order — per 09-UI-SPEC.md use `fixed top-20 right-4
z-50 flex flex-col gap-2`. Give it `x-data` holding two booleans initialised from the two PHP flags via
`@js(...)`, and an `x-init` that starts a ~4000ms `setTimeout` to hide the **status** toast only.

**The error toast must never auto-dismiss.** Do not add a timer to it. Per 09-CONTEXT.md an error the
user misses is precisely the FIX-03 bug this component exists to fix; an auto-hiding error re-creates
it. Comment the asymmetry so it reads as deliberate rather than an oversight.

Each toast: an inline SVG icon (left), the message text, and a close button (right). Follow the
codebase's existing icon convention — inline Heroicons-sourced SVG paths copy-pasted, no package (the
navbar's sun/moon toggle icons are the precedent). Use a check-circle icon for status and an
exclamation-circle for error.

Render the message text with Blade's escaping echo — `{{ $status }}` / `{{ $error }}`. **Never** use
the raw-output echo for flash text (T-09-02). Today every flashed value is a `__()`-wrapped literal,
but this component must not depend on that staying true. `ToastTest`'s
`test_the_toast_escapes_html_in_flash_text` pins it; comment the line with the threat id.

The close button is icon-only, so it **must** carry `aria-label="Dismiss"` — without it the control is
unlabelled for screen readers (09-UI-SPEC.md's checker recommendation, folded into the contract). Add
`type="button"` so it never submits a surrounding form.

Styling: reuse the UI-03 tokens ported in plan 09-06 — `rounded-base`, `shadow-xs`,
`border border-default`, `bg-neutral-primary-soft`, `text-body`. This is the token set's second
consumer after the login card. The error variant additionally takes a red accent (`border-red-300` or
equivalent from the existing app red scale — per 09-UI-SPEC.md's Color table, destructive red is NOT a
new token). Do NOT add `dark:` classes to the token-based surfaces; the tokens already resolve in both
themes through the `.dark` custom-property flip. The red accent, not being a token, does need its
`dark:` variant.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| controller flash → rendered HTML | `session('status')` / `session('error')` text crosses into the DOM. All 57 current call sites flash `__()`-wrapped literals, but the component must not assume that holds forever — a future caller could interpolate a model attribute or user input. |
| session store → guest shell | The guest shell now renders flashes on unauthenticated pages; the same escaping and exclusion rules apply there. |
