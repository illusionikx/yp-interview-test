Create the directory `app/Support/` and the file `app/Support/Semester.php` declaring
`namespace App\Support;` and `final class Semester`. PSR-4 already maps `App\` to `app/`, so no
composer.json change and no `composer dump-autoload` is needed. Import `Illuminate\Support\Carbon`.

**Constructor:** `public function __construct(public readonly int $year, public readonly int $number)`.
In the body, throw `new \InvalidArgumentException("Semester number must be 1 or 2, got {$number}.")`
unless `$number` is 1 or 2. Promoted readonly properties make the object immutable, which is the whole
point of a value object — no consumer can mutate a semester out from under another.

**Locked semantics — put this in the class-level doc-comment, not just in your head:**
- `Semester(Y, 1)` = 1 Sep Y → last day of Feb Y+1 (S1 crosses a calendar-year boundary)
- `Semester(Y, 2)` = 1 Mar Y → 31 Jul Y
- Therefore **S2-Y precedes S1-Y chronologically** (Mar Y is before Sep Y). The academic-year reading
  where S1-Y is followed by S2-(Y+1) is NOT this model. Settled by SEM-01's wording — "of the
  following year" qualifies only S1's February, while S2 is "March → July" unqualified — and by
  D-04, which fixes `Section.year`/`Section.semester` as the source of truth for the (year, number) pair.
- August (1-31) falls in NO semester's range, but `forDate()` still returns one (D-01 roll-forward).
  State this asymmetry explicitly: `forDate()` is a **total** function answering "which semester is
  current for this date"; `contains()` is a **range predicate** and is false for every August date.

**`public static function forDate(\Carbon\CarbonInterface $date): self`** — use a
`match (true)` on `$month = (int) $date->format('n')`, with this branch order (order is load-bearing):
- `$month >= 9` → `new self($date->year, 1)` — Sep-Dec belong to S1 of the same year
- `$month <= 2` → `new self($date->year - 1, 1)` — **Jan/Feb belong to the S1 that STARTED last
  September**, hence the `- 1`. Getting this wrong is 09-RESEARCH.md Pitfall 1 and silently corrupts
  every downstream `ordinal()` comparison.
- `$month <= 7` → `new self($date->year, 2)` — Mar-Jul are S2 of the same year
- `default` → `new self($date->year, 1)` — August only. D-01: roll FORWARD to the S1 starting this
  September. A null would blank every dashboard for a month. Comment it with the decision reference.

**`public static function current(): self`** — `return self::forDate(now());`.

**`public function startsAt(): Carbon`** — S1 → `Carbon::create($this->year, 9, 1)->startOfDay()`;
S2 → `Carbon::create($this->year, 3, 1)->startOfDay()`.

**`public function endsAt(): Carbon`** — S1 → `Carbon::create($this->year + 1, 2, 1)->endOfMonth()->endOfDay()`;
S2 → `Carbon::create($this->year, 7, 31)->endOfDay()`. Note the `$year + 1` on S1's end. Use
`endOfMonth()` for February and **never** a literal day number — a hardcoded 28 silently drops Feb 29
in a leap year (09-RESEARCH.md Pitfall 2). Comment that line to say so.

**`public function contains(\Carbon\CarbonInterface $date): bool`** — `$date->between($this->startsAt(), $this->endsAt())`,
or the equivalent `gte`/`lte` pair. Inclusive on both ends is correct here: `endsAt()` is already
end-of-day on the last day, and the semesters are non-contiguous (S2 ends 31 Jul, S1 starts 1 Sep), so
there is no adjacent boundary for a half-open interval to disambiguate — unlike
`Section::windowStatus()`, whose `[opens_at, closes_at)` windows do abut. Comment the divergence so a
reader does not "fix" it to match the sibling predicate.

**`public function ordinal(): int`** — `return $this->year * 2 + (2 - $this->number);`
This yields S2-Y → `2Y`, S1-Y → `2Y+1`, S2-(Y+1) → `2Y+2` — strictly increasing in `startsAt()` order.

**CORRECTION to 09-RESEARCH.md, do not skip this comment in the code:** research proposes
`$this->year * 2 + ($this->number - 1)`, which ranks S1-2026 below S2-2026. That contradicts
research's own `startsAt()`, where S2-2026 starts Mar 2026 and S1-2026 starts Sep 2026. Research's
proposed test only compared S2-2026 against S1-2027, so the inversion passed unnoticed. Adopting it
would break Phase 11's "this and future semesters" dashboard query — the precise SEM-02 failure the
requirement exists to prevent. Write a short doc-comment on `ordinal()` recording the correct relation
and the reason, and pointing at `test_ordinal_is_monotonic_with_start_date` as the guard.

**`isCurrent()` / `isFuture()` / `isPast()`** — compare `$this->ordinal()` against
`self::current()->ordinal()` with `===`, `>` and `<` respectively.

**`public function label(): string`** — `return "{$this->year} Semester {$this->number}";`

**Do NOT** import or reference `App\Models\Section` (or any Eloquent model) from this class. Keep
`app/Support/` free of model coupling. 09-CONTEXT.md scopes "anything that consumes the semester rule
beyond its own tests" out of this phase — the `Section` → `Semester` accessor belongs to Phase 11.
