space-separated RGB channels with no `rgb()` wrapper in the CSS variable itself, and Tailwind
substitutes `<alpha-value>` at build time. That indirection is what lets one class name resolve
differently in light and dark without a `dark:` prefix.

Keys to add under `colors`: `brand` (an object with `DEFAULT`, `strong`, `medium`, `soft`),
`'fg-brand'`, `heading`, `body`, `default`, `'default-medium'`, and `neutral` (an object with

functions that spread `theme('colors')`, so extending `colors` alone is what makes `border-default`,
`border-default-medium`, `ring-brand`, `ring-brand-medium`, `ring-brand-soft`, `text-fg-brand`,
`bg-brand-strong` and `placeholder:text-body` all resolve. Add a one-line comment recording that fact
and its source (verified against `tailwindlabs/tailwindcss` `stubs/config.full.js`, per 09-RESEARCH.md
§ Pattern 2) so nobody later "fixes" it by duplicating the color definitions.

Note on the `default` key: it is lowercase and therefore distinct from Tailwind's special uppercase
`DEFAULT` key (which generates the bare `.border` utility). `.border-default` and `.border` coexist;
this is not a collision.

Note on the `neutral` key: `theme.extend` deep-merges, so adding `primary-soft` / `secondary-medium`
extends Tailwind's stock `neutral-50…950` scale rather than replacing it. Verified: this repo
currently uses zero `neutral-*` utilities, so there is nothing to regress either way.

Keys to add under `borderRadius`: `base: '0.5rem'` **and `xs: '0.125rem'`**.

**`borderRadius.xs` is a CORRECTION to 09-UI-SPEC.md and 09-RESEARCH.md — do not omit it.** Both list
only `base`. But the login card uses `rounded-xs` on the remember-me checkbox at `.planning/v3.md`
line 17, and Tailwind 3's radius scale has no `xs` key (it runs `none`/`sm`/`DEFAULT`/`md`/`lg`/…;
`xs` is a Tailwind 4 addition, where it equals `0.125rem` — the value Tailwind 3 calls `sm`). Without
this key `rounded-xs` emits nothing and the checkbox silently loses its radius — the exact UI-03
silent-failure mode, on the very card UI-03 exists to make work. Add a comment on the line saying so.

Keys to add under `boxShadow`: `xs: '0 1px 2px 0 rgb(0 0 0 / 0.05)'` (Tailwind 3 calls this value
`shadow-sm`; Flowbite 4 / Tailwind 4 call it `shadow-xs`). `borderRadius` and `boxShadow` take plain
string values — they are not colors, so they need no CSS-variable indirection.

**Edit `resources/css/app.css`.** After the three existing `@tailwind` directives (they must stay
first), add a new `@layer base` block containing a `:root` rule and a `.dark` rule, each defining the
eleven `--color-*` custom properties with the light and dark channel triplets from 09-UI-SPEC.md's
resolved table. Keep the existing per-line hex comments from that table (e.g. the `blue-600` /
`#2563eb` annotation) — they are the only place the numeric triplets are human-readable.

Every property defined on `:root` must also be defined on `.dark`. A missing entry in the `.dark`
block means that token silently keeps its light value in dark mode — a failure with no error message.
Count both lists before moving on: eleven and eleven.

Do NOT touch the ~28 existing view files that carry `dark:` utility classes. Tokens are additive;
those views keep working untouched. 09-CONTEXT.md defers that migration to Phase 14 (FIX-02, UX-05).

nothing.
