Create `app/Exceptions/AttemptVanishedException.php` (create the `app/Exceptions/` directory if
`ls app/Exceptions/` shows it absent) declaring `namespace App\Exceptions;` and
`class AttemptVanishedException extends \RuntimeException`.

**Why a self-rendering exception rather than a `bootstrap/app.php` handler registration:** Laravel
calls a `render(Request $request)` method on the exception instance if one exists, before consulting
the global handler. Putting the response next to the exception keeps the whole INT-01 contract — when
it fires, and what the user sees — in one readable file, and avoids editing `bootstrap/app.php`, which
this phase otherwise has no reason to touch. Record this reasoning in a class doc-comment.

Add a class doc-comment explaining *why* this type exists, in this file's local convention of
long "explain the reasoning" comments (see `Attempt::lockAndFinalize()`'s own doc-comment as the
house style):
- It is raised when a locked re-read of an `attempts` row returns null — the row was deleted by a
  concurrent request between this request's route-model binding and its `lockForUpdate()` read
  (threat T-09-01, a TOCTOU race).
- It exists as a distinct type rather than a `return false` because `false` already means "already
  finalized by a racing request — an idempotent no-op", and the two need different user-facing
  outcomes: "here's your result" vs. "that attempt is gone". Conflating them is 09-RESEARCH.md Open
  Question 1; this is its resolution (Assumption A1, research's own recommendation).
- Today nothing deletes an in-progress attempt. Phase 10's attempt-reset is the first path that will.
  The guard lands first, deliberately.

Define a public class constant for the copy:
`public const MESSAGE = 'This exam attempt is no longer available. Please return to your exam list.';`
This is 09-UI-SPEC.md's Copywriting Contract value, verbatim. Exposing it as a constant lets the tests
and any future caller reference one source rather than re-typing the sentence.

Implement `public function render(\Illuminate\Http\Request $request): \Symfony\Component\HttpFoundation\Response`:
- If `$request->expectsJson()`: return
  `response()->json(['expired' => true, 'vanished' => true, 'message' => __(self::MESSAGE)], 422);`
  **Both keys are load-bearing.** `expired` is retained so the already-shipped client-side autosave
  handler — which branches on `expired` to drive auto-submit — keeps working unchanged against this
  new case (09-RESEARCH.md Assumption A2). `vanished` is the added discriminator so Phase 10 can give
  the reset case its own treatment without guessing. Comment both reasons on the line.
- Otherwise: return
  `redirect()->route('student.exams.index')->with('error', __(self::MESSAGE));`
  The `error` flash key is deliberate: it is the key the `<x-toast>` error variant renders (plans
  09-03 / 09-07), and per 09-UI-SPEC.md the error variant persists until manually dismissed rather
  than auto-dismissing — correct for a message the student must act on. Comment it.

Import `Illuminate\Http\Request`. Do not add a `report()` method — the default reporting behavior is
correct; this is a real, log-worthy condition, not noise to silence.
  </action>

  <action>
**Site 1 — `app/Models/Attempt.php`, inside `lockAndFinalize()`.**
Add `use App\Exceptions\AttemptVanishedException;` to the file's imports. Immediately after
`$locked = self::whereKey($this->id)->lockForUpdate()->first();` (line 140) and BEFORE
`$locked->setRelation('exam', $this->exam);`, insert a null guard that throws
`new AttemptVanishedException()`.

Write the guard's comment in this method's established style — a long comment explaining *why*,
directly above the line that needs it (match the density of the surrounding comments at lines 127-136
and 162-168). It must say: the row vanished under this transaction because a concurrent
delete/reset committed between this request's route-model binding and this `lockForUpdate()` read
(T-09-01). This is NOT the "already finalized by a racing request" case — that one is handled below by
the `status === 'in_progress'` check and correctly returns false as an idempotent no-op. A vanished
row cannot be synced back to `$this` (there is nothing to sync) and cannot be finalized, so the only
honest outcomes are a crash or a typed failure; INT-01 requires the latter.

Do NOT restructure the rest of the method. The in-memory sync at lines 169-171 is deliberately
skipped on this path because throwing exits the closure — that is correct: there is no authoritative
row to sync from, and leaving `$this` untouched is better than inventing state. Note this in the comment.

Note the enclosing `DB::transaction($callback, 3)`: the `3` is a deadlock-retry count. A thrown
`AttemptVanishedException` is not a deadlock, so it propagates immediately without retrying — which
is what we want (re-reading a deleted row three times finds it deleted three times). Do not change the
retry count. State this in the comment so a later reader does not "helpfully" wrap the throw.

**Site 2 — `app/Http/Controllers/Student/AttemptController.php`, inside `answer()`.**
Add `use App\Exceptions\AttemptVanishedException;` to the file's imports. Inside the
`DB::transaction(function () use ($attempt, $data) { ... })` closure, immediately after
`$locked = Attempt::whereKey($attempt->id)->lockForUpdate()->first();` (line 172), insert a null guard
that throws `new AttemptVanishedException()`.

Keep the existing `if ($locked->status !== 'in_progress') { return false; }` check exactly as it is,
as a separate statement below the new guard. Do NOT collapse the two into
`if (! $locked || $locked->status !== 'in_progress')` — 09-PATTERNS.md sketches that shape, but it
conflates two outcomes that must differ: a non-`in_progress` attempt is an ordinary expiry and must
keep returning false so the existing `422 {'expired': true, 'message': 'This attempt has ended.'}`
response at lines 186-192 fires unchanged, while a vanished row must produce the distinct
`vanished` response. Merging them would silently re-label every deleted attempt as "this attempt has
ended", which is the wrong message and defeats 09-UI-SPEC.md's copywriting contract. Put this
reasoning in the comment above the guard — it is the single most likely thing for a future editor to
"simplify" incorrectly.

Add a short comment noting that this read is INDEPENDENT of `lockAndFinalize()` — it is a second
lock site in its own transaction, which is exactly why guarding the model alone was insufficient
(09-RESEARCH.md Pitfall 5). A reader who fixes one site must know to check the other.

**Do not touch `show()` or `submit()`.** Both reach site 1 through `finalizeIfExpired()`/`finalize()`;
the exception propagates out of the controller and renders itself. No per-caller handling is needed —
that single-chokepoint property is the whole reason for the exception shape.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| student browser → `POST /student/attempts/{attempt}/answers` | Untrusted autosave crossing into a row a concurrent lecturer-side reset (Phase 10) may have just deleted |
| student browser → `GET /student/attempts/{attempt}` | Page-load touch of a possibly-vanished attempt |
| concurrent request → `attempts` row | The racing writer; the row can disappear between this request's route-model binding and its locked read |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |

Never assume the row exists because the route-model binding resolved it earlier in the request. | T-09-07 | Information Disclosure | `AttemptVanishedException::render()` | accept | The rendered message states only that the attempt is unavailable. Route-model binding + `AttemptPolicy` already 404/403 before this point; the exception is only reachable for an attempt the caller was authorized to touch. |

user input, and the escaping mitigation lives with the component in plan 09-07. T-09-03 does not apply.
