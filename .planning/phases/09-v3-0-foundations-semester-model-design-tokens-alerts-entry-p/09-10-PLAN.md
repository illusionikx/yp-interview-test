**Wrap `<x-modal>`; do not rebuild it.** The primitive already ships a complete, working overlay,

duplicate shipped code and introduce a second, divergent interaction pattern — the exact thing
09-RESEARCH.md's "Don't Hand-Roll" table forbids for this component.

Declare props first:
`@props(['name', 'title', 'body', 'confirmLabel' => 'Delete', 'cancelLabel' => 'Cancel', 'danger' => true])`

**Do not hardcode the three delete messages into this component.** 09-UI-SPEC.md is explicit that the
props exist so Phase 10's INT-02/CLS-07 warnings can pass a dynamic count as the body ("N students have

Body:
- `<x-modal :name="$name" max-width="md" focusable>` as the wrapper. Pass `focusable` — the primitive

  blocking decision dialog should take focus.

- Render `{{ $title }}` and `{{ $body }}` with Blade's escaping echo. Never the raw echo — the body

For each of the 3 destructive forms: remove the inline `onsubmit` attribute that calls the native
dialog, and replace it with an Alpine intercept plus a paired `<x-confirm-modal>`.

**Preserve the form-submit semantics exactly.** Keep `method="POST"`, the `action`, `@csrf`,
`@method('DELETE')`, and the submit button untouched. Introduce no `fetch`, no AJAX, no controller
change and no route change — the destructive request must still be an ordinary form POST to the same
endpoint. The only thing changing is what happens between the click and the submit.

Wiring per form:
- Add `@submit.prevent` to the `<form>` dispatching `open-modal` with the modal's key, and give the form
  a reference the modal's confirm button can reach so it can call `.submit()` on the right form. The
  exact mechanism (an `x-ref` within a shared `x-data` scope, or a small Alpine store keyed by the modal
  name) is your call per 09-UI-SPEC.md — what is fixed is that the underlying form actually submits on
  confirm.
- Note the DOM-node trap: call `.submit()` on the form **element**, not on a wrapped/proxied object, and
  be aware that a form containing a control named `submit` shadows the method. These forms do not, but
  keep the reference explicit rather than relying on `$el` resolving to the form from a nested handler.
- Pair each form with an `<x-confirm-modal>` carrying its resolved copy.

**Modal keys — these must be unique per rendered instance.** Two of the three forms are inside loops:
- delete exam → `delete-exam` (a single instance on the page; a plain key is fine)
- delete question → `delete-question-{{ $question->id }}` (one per loop iteration)
- delete subject → `delete-subject-{{ $subject->id }}` (one per loop iteration)
A shared static key inside a loop makes every row's modal open at once, because the primitive matches
`$event.detail` against `$name` and every instance would match. This is the single most likely bug in
this task and it will not be caught by the static-scan test — verify it by hand (see the acceptance
criteria).

Copy — use 09-UI-SPEC.md's Copywriting Contract verbatim:
- delete exam → title "Delete exam?", body "This permanently removes “{{ $exam->name }}” and all its
  questions. This cannot be undone."
- delete question → title "Delete question?", body "This permanently removes this question and its
  options. This cannot be undone."
- delete subject → title "Delete subject?", body "This permanently removes “{{ $subject->name }}”.
  Subjects with exams cannot be deleted."

Note the body strings contain curly typographic quotes around the interpolated name — that is
deliberate and matches the contract. Pass the body via a bound prop (`:body="..."`) so the interpolation
happens in PHP.

Set `confirm-label="Delete"` and leave `danger` at its default true for all three.

**Do not weaken the guard.** Removing the confirmation entirely would satisfy the scan for native
dialogs while deleting a destructive-action safeguard — `NoNativeDialogTest` test 2 exists precisely to
catch that, but do not rely on the test to keep you honest.

Do not re-add the inline flash banners plan 09-09 removed from these same two files.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| lecturer browser → destructive POST (`exams.destroy`, `questions.destroy`, `subjects.destroy`) | The confirmation is the last client-side guard before an irreversible, cascading delete. It is a UX guard, not a security control — the server-side policy is the actual authorization boundary and is unchanged. |
| Alpine client state → form submission | The modal's confirm handler triggers the submit; the form's CSRF token and method spoofing field are unchanged and still validated server-side. |
