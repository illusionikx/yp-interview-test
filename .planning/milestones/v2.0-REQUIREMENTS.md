# Requirements: Online Examination Portal

**Defined:** 2026-07-15
**Core Value:** A student can take a time-limited exam that is correctly restricted to their class, and their answers are reliably captured and scored.

> Authentication (register/login/logout/password reset/email verification/profile) is already provided by the Laravel 11 + Breeze scaffold and is treated as **Validated** in PROJECT.md — it is not re-listed as a v1 requirement. The requirements below are the net-new exam-portal domain built on top of Breeze.

## v1 Requirements

Requirements for initial release. Each maps to roadmap phases.

### Roles & Access Control

- [x] **RBAC-01**: System supports two roles — Lecturer and Student — stored on the user
- [x] **RBAC-02**: Public self-registration always creates a Student; Lecturer accounts are provisioned via seeding
- [x] **RBAC-03**: After login, a user is directed to a role-appropriate area (lecturer vs student)
- [x] **RBAC-04**: Lecturer-only pages and actions are inaccessible to Students (server-enforced by middleware, not just hidden in the UI)
- [x] **RBAC-05**: A Student can only access a resource (exam, attempt, result) belonging to them or their class — direct URLs to others' resources are denied (no IDOR)

### Class & Subject Management

- [x] **CLS-01**: Lecturer can create, edit, and delete classrooms (student groups)
- [x] **CLS-02**: Lecturer can create, edit, and delete subjects
- [x] **CLS-03**: Lecturer can associate multiple subjects with a classroom
- [x] **CLS-04**: Lecturer can assign a Student to a classroom

### Exam Authoring

- [x] **EXM-01**: Lecturer can create an exam that belongs to a subject, with a title and a time limit in minutes
- [x] **EXM-02**: Lecturer can add multiple-choice questions with multiple options and mark exactly one option correct
- [x] **EXM-03**: Lecturer can add open-text questions to an exam
- [x] **EXM-04**: Lecturer can set a per-question point value (default 1)
- [x] **EXM-05**: Lecturer can edit and delete an exam and its questions while it is unpublished
- [x] **EXM-06**: Lecturer can publish an exam to make it available to assigned classes (draft vs published)

### Exam Assignment

- [x] **ASN-01**: Lecturer can assign an exam to one or more classrooms
- [x] **ASN-02**: A Student sees only published exams assigned to their own classroom

### Taking Exams

- [x] **TAK-01**: Student can start an assigned exam, creating a single timed attempt
- [x] **TAK-02**: The exam time limit is enforced server-side; the UI shows a live countdown driven by the server deadline
- [x] **TAK-03**: Student answers are saved incrementally as they are entered (autosave), surviving a refresh or disconnect
- [x] **TAK-04**: When the time limit expires, the attempt auto-submits and further answer changes are rejected
- [x] **TAK-05**: A Student can attempt each exam only once (enforced by a database unique constraint)
- [x] **TAK-06**: Correct answers are never exposed to the Student while taking the exam

### Grading & Results

- [x] **GRD-01**: Multiple-choice answers are auto-graded on submission
- [x] **GRD-02**: Lecturer can grade open-text answers, assigning a score up to the question's point value
- [x] **GRD-03**: A Student's final result is shown only after all open-text answers in the attempt are graded
- [x] **GRD-04**: Student can view their own graded result — total score plus a per-question breakdown
- [x] **GRD-05**: Lecturer can view results per exam and per student

### Demo & Delivery

- [x] **DEL-01**: A seeder creates demo data (lecturer, students, classrooms, subjects, and a sample exam with both question types assigned to a classroom), runnable via `php artisan migrate:fresh --seed`
- [x] **DEL-02**: README documents setup, database creation, migration, seeding, demo credentials, and how to run/test the app

## Milestone v2.0 Requirements

Net-new scope for milestone v2.0 (Enrollment, Exam Availability & Fixes). Replaces the v1 single-classroom model with a subject-scoped, capacity-and-window-gated enrollment system; adds exam availability windows + a pre-start page; adds an admin UI theme with dark mode; adds non-technical user manuals; fixes the answered-count bug. **Schema break:** original v1 migrations are edited in place — `php artisan migrate:fresh --seed` is required.

### Class Sections & Subject–Lecturer

- [x] **SEC-01**: A subject can have multiple class-sections; each section belongs to exactly one subject and is identified as `year-semester-count` (semester is 1 or 2, representing the half-year), e.g. `2026-2-1`
- [x] **SEC-02**: Lecturer can create, edit, and delete a section, setting its capacity and an enrollment window (open date, close date)
- [x] **SEC-03**: A subject can be assigned to one or more lecturers; any assigned lecturer can manage that subject's sections, enrollments, and exams

### Student Enrollment

- [x] **ENR-01**: Student can choose a subject and see its sections with live capacity (e.g. `28/30` or `FULL`) and enrollment-window status
- [x] **ENR-02**: Student can apply to a section and is enrolled immediately (no approval step) when the enrollment window is open and the section is not at capacity; concurrent applies can never oversell a section's capacity
- [x] **ENR-03**: Student can withdraw from a section before its enrollment close date
- [x] **ENR-04**: A student can hold at most one **active** enrollment per subject **per semester** (i.e. one active section among a subject's sections that share the same year+semester term)
- [x] **ENR-05**: A student who has withdrawn from or been rejected from a section may re-apply while the enrollment window is still open (neither withdrawal nor rejection is permanent)
- [x] **ENR-06**: Sections outside their enrollment window remain listed with a status label ("Opens …" / "Closed") and cannot be applied to
- [x] **ENR-07**: Any lecturer of the subject can reject an enrolled student, choosing a reason from a fixed dropdown (not free text); the rejected student can see the reason
- [x] **ENR-08**: A student sees and can take only published exams assigned to a section in which they are actively enrolled; the exam list and the direct-access gate use one shared rule (no divergence, no IDOR)

### Exam Availability, Pre-Start & In-Progress

- [x] **AVL-01**: Lecturer can set an availability window on an exam — a start and/or an end datetime, each optional (empty = unbounded on that side)
- [x] **AVL-02**: Before starting, a student sees a pre-start exam details page (instructions, duration, availability window, their section) with Proceed and Back actions
- [x] **AVL-03**: A student can start an attempt only within the availability window; outside it, starting is refused with a clear "not yet open" / "closed" message
- [x] **AVL-04**: An in-progress attempt is governed only by its own timer — the availability window and any later withdrawal or rejection never cut a started attempt short
- [x] **AVL-05**: While an attempt is in progress, attempting to close the tab/window or navigate away triggers a browser confirmation warning; the warning is suppressed during the intentional submit and the timer's auto-submit

### Admin Theme & Dark Mode

- [x] **UI-01**: A consistent admin-style shell (sidebar navigation + top bar + card content, with status pills for enrollment/exam/section states) is applied across the student and lecturer areas, built on Tailwind + Flowbite
- [x] **UI-02**: A light/dark mode toggle in the top bar persists the user's choice (localStorage) and defaults to the OS preference

### Fixes

- [x] **FIX-01**: The submit-confirmation "N of M answered" count reflects answers saved during the current session, not only those present when the page loaded

### Demo Data, Delivery & User Docs

- [x] **DEL-03**: The seeder and factories are rewritten for the section/enrollment model (sample sections named e.g. `2026-2-1`, sample enrollments, subject↔lecturer assignments, and exam availability windows) so `php artisan migrate:fresh --seed` stands up a working demo; the README notes the schema break requires `migrate:fresh`
- [x] **DEL-04**: A non-technical **student** user manual (plain language, no setup/CLI) covers: enrolling in a subject's section, withdrawing, seeing exam availability, taking a timed exam, and viewing results
- [x] **DEL-05**: A non-technical **lecturer** user manual (plain language) covers: managing subjects/sections, viewing rosters and rejecting students with a reason, authoring/assigning exams with an availability window, and grading

### Resolved Design Decisions (v2.0)

Open questions raised during research, resolved here using the researchers' recommendations (adjusted per user review). Boundary/technical items are documented for the phase plans.

| # | Decision | Resolution | Requirement |
|---|----------|-----------|-------------|
| 1 | Rejection reason dropdown values | `Not eligible for subject` · `Prerequisite not met` · `Duplicate enrollment` · `Section changed` · `Other` (fixed enum, server-validated) | ENR-07 |
| 2 | Does the student see the rejection reason? | Yes — visible to the rejected student | ENR-07 |
| 3 | How many active enrollments per subject? | At most one **active** section per subject **per semester** (year+half-year term) | ENR-04 |
| 4 | Can Withdrawn / Rejected students re-apply? | Yes — **both** may re-apply while the window is open (neither is terminal) | ENR-05 |
| 5 | Out-of-window sections stay listed? | Yes — shown with a status label, not hidden | ENR-06 |
| 6 | Window boundary semantics | Half-open `[opens_at, closes_at)` and `[available_from, available_until)` — allowed while `open ≤ now < close`; a null bound is unbounded on that side; tested at the exact boundary | SEC-02, AVL-01/03 |
| 7 | Post-start attempt access decoupled from live enrollment? | Yes — attempts are ownership-gated (like results), not visibility-gated; the enrollment/availability check applies at start only | AVL-04, ENR-08 |
| 8 | `subject_user` on-delete behavior | Cascade (consistent with existing pivot conventions) | SEC-03 |

## Out of Scope

Explicitly excluded. Documented to prevent scope creep.

| Feature | Reason |
|---------|--------|
| Real-time proctoring / webcam monitoring / AI cheat detection | Disproportionate complexity for the assessment scope |
| Question bank import/export, rich media in questions | Not core to exam-delivery value |
| Public lecturer self-registration | Self-signup to a privileged grading role is a security/integrity risk |
| SPA / separate API frontend (Livewire/Inertia/React) | Breeze Blade + Tailwind + Alpine is already scaffolded and sufficient |
| Live / multiplayer / collaborative exams | Single-taker exams only |
| Scheduled auto-submit sweep job (queue/cron) | Lazy server-side deadline check on every write is sufficient at this scale |
| Waitlists, capacity auto-promotion, enrollment history/audit, bulk import, section transfer | v2.0 enrollment is intentionally minimal (immediate-enroll, no approval); these are not baseline for this scope |
