## v3.0 Requirements

### Semester Model (SEM)

- [x] **SEM-01**: A semester is derived from a date, not stored as a table — 1st semester runs **September → February of the following year**; 2nd semester runs **March → July**. A semester always starts on the 1st day of its first month and ends on the last day of its last month.
- [x] **SEM-02**: Semesters have a total ordering that is correct across the S1 year rollover (Sep 2026 → Feb 2027 sorts as one semester, before Mar 2027), so "this semester and future semesters" is computable and past semesters can be identified.
- [x] **SEM-03**: The August gap (Aug 1–31, which belongs to neither semester by the stated rule) has a defined, tested "current semester" policy.

### Integrity Foundations (INT)

- [x] **INT-01**: `Attempt::lockAndFinalize()` null-guards its locked row, so an attempt deleted or voided concurrently cannot crash a racing timer/autosave request. *(Latent today; v3's attempt-reset is the first code path that can trigger it.)*
- [x] **INT-02**: Cancelling/resetting attempts never destroys a graded result silently — the lecturer is warned with the concrete count of attempts (and graded results) affected before the operation proceeds.
- [x] **INT-03**: A student whose attempt was cancelled/reset can take the exam again — the single-attempt constraint does not permanently lock them out.
- [x] **INT-04**: No exam is ever visible or takeable to a student enrolled only in a **different subject's** class — enforced structurally, not by UI filtering alone. *(Guards against re-opening v2.0's CRITICAL cross-subject leak when assignment becomes automatic.)*

### Navigation & Entry (NAV)

- [x] **NAV-01**: A landing page renders before login, replacing the Laravel/Breeze default.
- [x] **NAV-02**: The login page follows the Flowbite design supplied in `v3.md` (card, email, password, remember-me, lost-password, submit, create-account link).
- [x] **NAV-03**: Navigation follows the v3 hierarchy: `login → dashboard | class enrollment (student) | subject list (+CRUD lecturer) | subject & class management (lecturer) → {class & student management, exam management → {exam editor, grading}} | class (student) → exam list → take exam`.
- [x] **NAV-04**: Every shipped feature reachable by a role remains reachable by clicking after the restructure — no feature is orphaned behind a removed link. *(v2.0 lesson: existence ≠ reachability.)*

### Dashboard (DASH)

- [x] **DASH-01**: After login the landing page shows a dashboard at the top and the subject list beneath it, on the same page.
- [x] **DASH-02**: The dashboard shows a welcome banner with a gradient treatment.
- [x] **DASH-03**: A lecturer sees cards for: total classes assigned this and future semesters; total students enrolled vs. total seats across all assigned classes (with a progress bar); plus at least one further relevant stat.
- [x] **DASH-04**: A student sees cards for: total subjects enrolled this semester; plus at least one further relevant stat.

### Subject List (SUBJ)

- [x] **SUBJ-01**: A lecturer sees every subject assigned to them in a table (no grouping), with an action to open subject & class management.
- [x] **SUBJ-02**: A lecturer can create, edit, and delete subjects (simple CRUD).
- [x] **SUBJ-03**: A student sees subjects grouped by semester in a table, each showing subject details including lecturer name, with an action to open the class page.
- [x] **SUBJ-04**: A student can hide/unhide past semesters in the subject list.
- [x] **SUBJ-05**: A student has a button to enroll in a class from the subject list.

### Class Enrollment (ENR)

- [x] **ENR-09**: Class enrollment is a single page with a clear select subject → select class → enroll flow.
- [x] **ENR-10**: A student may enroll in as many subjects as they want — no credit limit.
- [x] **ENR-11**: A student can only enroll in a class that is currently open for enrollment.

### Class Management (CLS)

- [x] **CLS-01**: Subject & class management shows two tabs; the class list is the default tab and the exam list is the second.
- [x] **CLS-02**: The classes tab lists classes grouped by semester, with hide/unhide for past semesters; each row shows class code and total/max students as a progress bar, plus actions.
- [x] **CLS-03**: A lecturer can create, edit, and delete a class; the form includes max students, location, and enrollment period.
- [x] **CLS-04**: The exams tab lists all exams for the subject with create/edit/delete.
- [x] **CLS-05**: Every student enrolled in the subject's classes is automatically assigned to every active exam in that subject — no manual per-class exam assignment step.
- [x] **CLS-06**: A lecturer can toggle an exam between draft and active (inactive), both directions.
- [x] **CLS-07**: A lecturer can reset an exam's submissions, behind a warning that states what will be lost.
- [x] **CLS-08**: The exams tab shows grading progress per exam.

### Exam Editor (EDT)

- [x] **EDT-01**: The exam editor carries an exam/test name.
- [x] **EDT-02**: Exam details and questions live on one page as two tabs.
- [x] **EDT-03**: Answer options can be reordered, with a single option to shuffle/randomize their arrangement at authoring time.
- [x] **EDT-04**: Saving editor changes cancels all previous student attempts; if any student has attempted previously, a warning pops up first.
- [x] **EDT-05**: Each question shows its number with move-up / move-down buttons on its left.

### Grading (GRD)

- [x] **GRD-06**: The grading page shows class details, exam details, and grading progress, and lists every student.

### Class Page & Take Exam (TAK)

- [x] **TAK-07**: The class page shows subject detail plus an exam list in a separate card, each exam marked not-yet-open / opened / closed, and whether it has been taken or graded.
- [x] **TAK-08**: A student can take each exam only once per subject per class — the start button is disabled to block reaching the exam start page again.
- [x] **TAK-09**: The take-exam page has a sticky top bar with exam title, timer, and question progress, plus a button that opens the exam instructions in a popup; the header carries subject name and exam/test name with exam details and instructions separated.
- [x] **TAK-10**: Question navigation uses a vertical stepper with a checkmark on answered questions, and each question shows its number on the left.
- [x] **TAK-11**: When the timer reaches 10 minutes remaining a toaster appears and the timer turns red.
- [x] **TAK-12**: Answer options render in their authored arrangement — never randomized at take time.

### UI System (UX)

- [x] **UX-01**: The app is titled **"Online Examination Portal"** with the subtitle *"for Yayasan Peneraju Technical Assessment"*.
- [x] **UX-02**: One popup/alert style is used throughout; the browser's native `alert()` is never used.
- [x] **UX-03**: A toaster appears on create, save-after-changes, and delete across the app.
- [x] **UX-04**: Navigation-back affordances are buttons with text that names where they lead — not bare links.
- [x] **UX-05**: A help button sits beside the light/dark toggle and opens the user manual.
- [x] **UI-03**: Flowbite 4's semantic design tokens (`bg-brand`, `text-heading`, `bg-neutral-primary-soft`, `rounded-base`, …) resolve to real CSS under the project's Tailwind 3 setup. *(Verified blocker: those tokens ship in a Tailwind-v4-only `@theme{}` block and currently emit NO CSS — an unstyled page, not a build error.)*

### User Manual (DEL)

- [x] **DEL-06**: The user manual is wiki-style — topic navigation with cross-links, rather than v2.0's single linear page per role.

### Demo Data (SEED)

- [x] **SEED-01**: The seeder creates many lecturers and students with unique, realistic names; only lecturers carry Dr/PhD prefixes.
- [x] **SEED-02**: The seeder includes past-semester data consistent with the workflow — graded exams and filled classes exercising every available status.
- [x] **SEED-03**: The seeder adds 3–5 further subjects, classes, and exams.
- [x] **SEED-04**: `php artisan migrate:fresh --seed` still stands up the full demo from an empty database. *(Non-negotiable — the documented clean-clone path.)*

### Browser Testing (TEST)

- [x] **TEST-01**: Laravel Dusk is installed and runs against Herd (`APP_URL=http://yp-test.test`, no `artisan serve`).
- [x] **TEST-02**: Dusk uses a **separate** database via `.env.dusk.local` + `DatabaseTruncation` — it never truncates the documented `yp-student-exam` dev database.
- [x] **TEST-03**: Dusk covers the primary student and lecturer end-to-end flows.
- [x] **TEST-04**: The existing PHPUnit suite (294 passing at v2.0 close) stays green alongside Dusk.

### Bugfixes (FIX)

- [x] **FIX-02**: Dark-mode text contrast is correct on the exam editor and site-wide — no text left dark-on-dark. *(416 `dark:` occurrences across 28 files with no token layer; needs a systematic sweep, not per-view patching.)*
- [x] **FIX-03**: "Update assignment" gives clear feedback rather than appearing to return to the same page with nothing changed. *(Satisfied by removal, Phase 10/D-1: the manual exam-assignment screen — controller, request, route, view panel, and tests — was deleted outright once CLS-05 made visibility subject-derived. The bug cannot exist because the button that caused it is gone; no toaster was built for a deleted screen.)*

## Resolved Design Decisions (v3.0)

Open questions raised during research, resolved here. **Autonomously decided while the user was away** ("handle decision by yourself since i'll be away") — each is flagged for override.

| # | Decision | Resolution | Requirement |
|---|----------|-----------|-------------|
| 1 | August gap — what is "current semester" between Jul 31 and Sep 1? | **Roll forward** to the upcoming semester. A null would blank every dashboard for a month. | SEM-03 |
| 2 | Editor "shuffle" — authoring-time or per-student runtime? | **Authoring-time only.** v3.md's own take-exam rule says "dont randomize answer sequence, follow arrangement" — runtime shuffle is excluded by the user's words. | EDT-03, TAK-12 |
| 3 | "Class" vs. the shipped `Section` entity | **UI-copy relabel only — do NOT rename `Section` again.** `Section::name` is already a computed accessor; a second in-place rename after v2.0's would touch every FK/factory/test for zero functional gain. Route names stay too. | NAV-03, CLS-* |
| 4 | Semester — stored table or derived? | **Derived value object** (`App\Support\Semester`). `Section.year`/`semester` are already the source of truth. Needs an `ordinal()` total order for the S1 rollover. | SEM-01, SEM-02 |
| 5 | Flowbite 4 tokens under Tailwind 3 | **Port the token *values* into `tailwind.config.js` as v3 `theme.extend`** — do NOT upgrade to Tailwind v4. Keeps the user's login design exactly, without a framework upgrade rippling through 28 files of `dark:` classes. | UI-03 |
| 6 | Does Dusk make v2.0's deferred `beforeunload` UAT automatable? | **No.** ChromeDriver 126+ auto-dismisses the native prompt before Dusk's dialog API sees it. AVL-05 stays a manual check. *(This corrects an earlier orchestrator assumption.)* | TEST-03 |
| 7 | Dusk database | **Separate DB via `.env.dusk.local` + `DatabaseTruncation`** — never `yp-student-exam` (Dusk runs a real HTTP process, so `RefreshDatabase` cannot work and the default would wipe the demo data). | TEST-02 |
| 8 | Answer reordering mechanism | **Move-up / move-down buttons**, zero new dependency and Dusk-testable — consistent with the question-reorder UX v3.md already asks for. Drag-and-drop deferred. | EDT-03, EDT-05 |
| 9 | Navbar — "might be unnecessary" | **Keep a slim top bar.** v3.md itself asks for a help button "nearby those light dark toggle", so the bar must survive to host them; navigation moves into the in-page hierarchy. | NAV-03, UX-05 |

### Deferred to discuss-phase (NOT decided here)

These are genuine design decisions with real blast radius. Research deliberately declined to resolve them, and they must be settled in their phase's discuss step **before** dependent work.

**All three land in Phase 10** — `/gsd-discuss-phase 10` must resolve them before Phase 10 is planned.

| Question | Why it's deferred | Requirement | Phase |
|----------|-------------------|-------------|-------|
| **Auto-assignment scope** — does CLS-05 drop the `exam_section` pivot and derive visibility from subject enrollment, or keep the pivot and auto-populate it? | **Highest-risk decision in v3.** Dropping the pivot makes v2.0's cross-subject leak structurally impossible; keeping and auto-populating it **re-opens that exact CRITICAL leak** unless every write is scoped by `exam.subject_id === section.subject_id`. One decision, two safe paths, one unsafe path. | CLS-05, INT-04 | 10 |
| **Attempt cancel/reset mechanism** — reset-in-place, hard delete, or a schema change to the unique key? | Collides with three shipped invariants: `UpdateExamRequest::authorize()` forbids editing a published exam at all; `answers.attempt_id` is `cascadeOnDelete()` (hard delete destroys graded scores); `attempts.unique(exam_id,user_id)` blocks the retake a reset is meant to enable. An orchestrator "soft-void" guess was retracted for exactly this reason. | INT-02, INT-03, CLS-07, EDT-04 | 10 |
| **Reset-submission granularity** — per-exam (v3.md's placement) or also per-student? | v3.md puts the action in the exams list, implying per-exam; per-student is the more common real need. | CLS-07 | 10 |

- **Renaming `Section` → `Class`** — UI relabel only (Decision #3).
