# Phase 5 — UI Design Contract

> Visual and interaction contract for the lecturer open-text grading screen, the lecturer per-exam results index, and the gated student result view. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> This phase's one hard rule is a security/integrity rule, not a cosmetic one: **the student result view must never expose the correct MCQ answer key, and must never render a score of any kind while `attempt.status !== 'graded'`** (D-05, D-07, 05-RESEARCH.md Pitfall 1 & 3). Every visual rule below is subordinate to the locked backend contract in `05-CONTEXT.md` (D-01..D-08) and `05-RESEARCH.md`. This contract deliberately extends, rather than replaces, `04-UI-SPEC.md`'s tokens — same spacing scale, same two font weights, same 60/30/10 palette — because Phase 5's screens are the direct continuation of the take-exam flow the student and lecturer already saw in Phase 2–4.

This phase extends it with a `✗`/`text-red-700` pairing for "incorrect," since — unlike the authoring view — the results/grading views must represent both outcomes, not just flag which single option is correct.
- Table pattern (already established, `lecturer/exams/index.blade.php`): `min-w-full divide-y divide-gray-200`, `th` cells `px-4 py-2 text-left text-xs font-medium text-gray-500 uppercase tracking-wider`, `td` cells `px-4 py-2 text-sm`.
- Focus/accent: `indigo-500`/`indigo-600` (focus rings, links, `@tailwindcss/forms` native input accent color)
- Meta/muted text: `text-gray-500`, `text-sm`

This phase does not introduce a new palette — it reuses the established one and adds the minimum new semantic states (attempt status, MCQ/open-text correctness, grading progress) documented below.

## Spacing Scale

Declared values (must be multiples of 4) — identical to `04-UI-SPEC.md`'s scale, no new tokens introduced:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Gap between the `✓`/`✗` glyph and its "Correct"/"Incorrect" label text; gap inside a status badge |
| sm | 8px | Gap between table cells' inline content (e.g. score + "Grade" link); gap between the point-value badge and question body |
| md | 16px | Padding inside a question/answer card (`p-4`) on narrow layouts; table cell horizontal padding (`px-4`, matches existing `lecturer/exams/index.blade.php`) |
| lg | 24px | Card padding on grading/results pages (`p-6`, matches existing `show.blade.php`/`index.blade.php`); gap between answer cards |
| xl | 32px | Vertical rhythm between the attempt-status header block and the first answer card on the lecturer grading screen |
| 2xl | 48px | Space above/below the total-score block on the student result page |
| 3xl | 64px | Vertical centering offset on the "awaiting grading" student result state (mirrors the Submitted Confirmation page's calm, centered treatment from Phase 4) |

Exceptions:
- **Score input width is fixed, not fluid**: the open-text grade `<input type="number">` on the lecturer grading screen is `w-24` (96px) regardless of viewport — a numeric 0–N score never needs to grow, and a fixed narrow width visually communicates "small bounded number" next to the answer text it belongs to.
- **44px minimum touch target** on every "Save Score" button — same rationale as Phase 4's Submit button (a mis-tap on a grading action affecting a real student's score should not be easy), `py-2 px-4` minimum, reusing `x-primary-button`'s existing sizing (already meets this).

### Status Colors (phase-specific extension — attempt status & correctness only)

The base 60/30/10 table above doesn't cover the three stateful indicators this phase requires (attempt status badge, MCQ correctness, open-text score-awarded badge). These are semantic status colors, scoped exclusively to the components named, and reuse the exact badge pattern already established in `lecturer/exams/index.blade.php`/`show.blade.php` (`bg-{color}-100 text-{color}-800`, `rounded-full`):

| State | Color | Applies to |
|-------|-------|------------|
| Attempt status — Submitted (pending grading) | `bg-amber-100 text-amber-800` | Lecturer results index status column + lecturer grading-screen header badge, while `attempt.status === 'submitted'` |
| Attempt status — Graded | `bg-green-100 text-green-800` | Same locations, once `attempt.status === 'graded'` |
| MCQ correctness — Correct | `text-green-700 font-medium`, glyph `✓` | Lecturer grading screen's read-only MCQ answer row; student result breakdown's MCQ row |
| MCQ correctness — Incorrect | `text-red-700 font-medium`, glyph `✗` | Same two locations, when `answer.is_correct === false` (including the "never answered" case, which auto-grades to incorrect per D-01) |
| Open-text score badge | `bg-gray-100 text-gray-800`, `rounded-full text-xs font-medium` | Displays "{score} / {points} pts" next to a graded open-text answer — neutral, not green/red, since open-text has no binary correct/incorrect (05-RESEARCH.md Open Question 1: `is_correct` stays `null` for open-text; render score only, no ✓/✗) |
| Grading progress bar | Track `bg-gray-200`, fill `bg-indigo-600`, `rounded-full h-2` | Lecturer grading screen only — "{X} of {N} open-text answers graded" |

## Screen Inventory & States

Three screens cover this phase's entire surface. The student result screen has two mutually-exclusive states gated on `attempt.status`, mirroring how Phase 4 gated its take-page on `in_progress` vs redirecting straight to the Submitted Confirmation screen — same "no in-between/partial view" discipline applies here.

### 1. Lecturer Grading Screen (`lecturer/results/show.blade.php` — also the per-attempt breakdown view, D-06)
- **Layout:** single column, `max-w-3xl mx-auto`, matches the existing `lecturer/exams/show.blade.php` container width.
- **Header block** (`bg-white rounded-lg shadow-sm p-6`): student name + exam title (Heading token); attempt status badge (Status Colors) top-right; grading progress bar + "{X} of {N} open-text answers graded" label directly below the heading, visible only while `status === 'submitted'` (hidden once `graded` — nothing left to track).
- **Answer list:** one card per question (`bg-white rounded-lg shadow-sm p-6`, `space-y-6` between cards), ordered by `question.position` (same ordering convention as the take-page and authoring views):
  - Meta line (Label token, `text-gray-500`): "Question {n} of {total} · {points} {points === 1 ? 'point' : 'points'}"
  - Question body (Body token, `font-semibold` — same authoring-consistent emphasis exception as Phase 4)
  - **MCQ row (read-only, always auto-graded already):** the student's selected option's text (Body token); correctness label (`✓ Correct` / `✗ Incorrect`, Status Colors); "Auto-graded" tag (Copywriting). Since this is lecturer-only (not the student-facing D-07 restriction), the row MAY additionally show a muted secondary line "Correct answer: {option body}" (`text-gray-500 text-sm`) when the student got it wrong — this helps the lecturer sanity-check the auto-grade without needing to reopen the exam's authoring page. If the student left the question unanswered (no `Answer` row), render "No answer submitted" (`text-gray-500 italic`) in place of the selected-option text, with correctness still shown as `✗ Incorrect` per D-01.
  - **Open-text row, ungraded:** the student's `answer_text` rendered verbatim in a bordered read-only block (`bg-gray-50 rounded-md p-4 text-sm text-gray-800 whitespace-pre-wrap` — preserves the student's line breaks); below it, a small inline form: a numeric score `<input type="number" min="0" :max="points" step="1" class="w-24 ...">` + "Save Score" button (`x-primary-button`, small size) + `x-input-error` slot for the bounds-validation message. If the student never touched this question (no `Answer` row), render "No answer submitted" in place of the answer block and still offer the score input (defaults to 0, lecturer can save 0 directly — matches D-01/Pitfall-2's "no row = 0 contribution" only applying to the auto-grade path, not blocking a lecturer from explicitly recording it).
  - **Open-text row, already graded:** same read-only answer block, but the score input row is replaced by the neutral open-text score badge ("{score} / {points} pts") plus a small "Edit" link that reveals the same inline form in-place (regrade path, 05-RESEARCH.md Open Question 2) — no separate page/route needed, just a local Alpine toggle.
- **No page-level "Save all" action** — each open-text answer saves independently (matches the per-question autosave discipline from Phase 4, and means a lecturer grading a long attempt never loses work already saved if they navigate away partway through).

### 2. Lecturer Results Index (per exam, `lecturer/results/index.blade.php`, D-06)
- **Layout:** `max-w-7xl mx-auto`, matches the existing `lecturer/exams/index.blade.php` container width and its exact table pattern (reused verbatim, not reinvented).
- **Header:** exam title (Heading token) in the page's `x-slot name="header"`, same convention as every other lecturer index/show page.
- **Table columns:** Student (name), Status (attempt-status badge), Score (Display-adjacent, smaller: "{score} / {total}" once graded, "—" while submitted-and-pending — never a partial number, mirrors Pitfall 3's rule applied to the list view too), Action (`Grade` / `View` per Copywriting).
- **Empty state:** table renders a single full-width row with the "No submissions yet" copy (matches the existing `colspan` empty-row pattern in `lecturer/exams/index.blade.php`), not a separate empty-state screen.
- **Entry point:** reached via the new "View Results" link added to `lecturer/exams/show.blade.php`'s existing action row (next to Publish/Unpublish/Edit/Delete).

### 3. Student Result View (`student/results/show.blade.php`, D-05) — two gated states
- **Layout (both states):** centered card, `max-w-md mx-auto` for the awaiting state / `max-w-2xl mx-auto` for the graded breakdown state (wider, since it holds a per-question list — matches the width jump pattern already established between Phase 4's `max-w-md` confirmation page and its `max-w-3xl` take page).
- **State A — Awaiting grading** (`status !== 'graded'`): calm, minimal card (`bg-white rounded-lg shadow-sm p-8 text-center`, `3xl` token vertical offset), no score data anywhere in the rendered HTML or any embedded JSON — heading + body per Copywriting, "Back to my exams" link. This state must be indistinguishable in structure from Phase 4's Submitted Confirmation page (same calm, single-card, no-header treatment) so a student doesn't perceive it as a different or broken page — it is the natural next chapter of the same "your exam is being handled" story.
- **State B — Graded** (`status === 'graded'`): header block (`bg-white rounded-lg shadow-sm p-6`) with "Your Result" heading + total score line (Display token, "{score} / {total} points"); below it, the per-question breakdown list — same card-per-question visual rhythm as the lecturer's grading screen (question meta, question body, the student's own answer, correctness/score per Copywriting) but **strictly read-only, no inputs, no correct-option text ever rendered** (D-07 — the one hard constraint of this whole phase). "Back to my exams" link at the bottom.

## Interaction Contract

These rules translate the D-01..D-08 backend contract into concrete UI behavior. They are non-negotiable for this phase — a checker/auditor should treat any deviation as a FAIL, not a style nit.

1. **The student result view never renders a score while `status !== 'graded'`.** Not zero, not a partial sum, not a "some questions graded" progress indicator — nothing numeric. The controller must not even pass score data into the view when withholding (05-RESEARCH.md Pitfall 3); this is a view-model contract, not just a template `@if`.
2. **The student result breakdown never renders the correct option's identity/text for a question the student got wrong.** No option markup, no muted "the answer was X" line, no title/alt-text attribute may carry it (D-07). A checker should grep the rendered student-result HTML for any `Option::where('is_correct', true)` text leak — this is the direct continuation of Phase 4's `is_correct`-in-markup grep rule, now applied to the *post-grading* surface instead of the *pre-grading* one. The lecturer-facing grading screen is explicitly exempt from this rule (see Screen 1's "Correct answer:" line) — the restriction is student-facing only.
3. **Every grading score input is bounded client-side to mirror the server**: `min="0"` and `max="{{ $question->points }}"` on the `<input type="number">`, so a lecturer gets instant feedback before the round-trip — but the server's `GradeAnswerRequest` bound remains the sole source of truth; the client bound is a UX nicety, never assumed sufficient on its own.
4. **A failed grade-save never blocks grading other answers.** The failure (bounds error or network error) is scoped to that one answer's inline error slot; the lecturer can keep grading the rest of the attempt.
5. **The grading progress label and bar are server-computed, not client-derived.** "{X} of {N}" comes from the page's initial render (or a full page reload after a save — no client-side optimistic recount), so it can never drift from the actual `syncStatus()` pending-count the backend uses to decide the `submitted → graded` transition (05-RESEARCH.md Pattern 2/Pattern 3).
6. **No delete/destructive action exists anywhere in this phase's UI.** Saving a score (first grade or regrade) is the only write action; there is no `x-danger-button` usage, no confirm-modal, no "are you sure" pattern needed (contrast with Phase 4's Submit confirm-modal, which existed because submission is irreversible — a grade save is not).
7. **Accessibility baseline:** every score input has a programmatically-associated `<label>` (visually can be the "Score" text preceding the input, or `sr-only` if the surrounding question card already provides enough context — either is acceptable, but never an unlabeled bare `<input>`); focus is visible on every interactive element (inherited Breeze default focus-ring, do not `outline-none` without a replacement ring); the grading-progress bar carries `role="progressbar"` with `aria-valuenow`/`aria-valuemax` so a screen-reader user gets the "{X} of {N}" fraction without needing to parse the visual bar; the `✓`/`✗` correctness glyphs are always paired with a text label ("Correct"/"Incorrect"), never the glyph alone, so the distinction isn't color/glyph-only (WCAG 1.4.1 — don't rely on color alone, matches the existing text-plus-color pattern already used for the Published/Draft badges).
