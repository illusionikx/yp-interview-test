<objective>
Ship the lecturer rejection surface: a section roster page listing enrolled students, and a
per-row Reject action gated by SEC-03 per-subject ownership and validated against the fixed
five-value reason enum — with the reason visible to the rejected student.

Purpose: completes ENR-07, the last enrollment-mutation path. Like 08-04's withdraw, this is a
code path that flips a student's enrollment away from Enrolled, so it depends on 08-03's
AttemptPolicy fix already having landed — rejecting a student mid-exam must not cut their
attempt short.

Output: a roster page, one new controller, one Form Request, two routes.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| lecturer browser -> GET /lecturer/sections/{section} | An arbitrary section id from any authenticated lecturer — the roster exposes other students' identities, so subject ownership must gate it. |
| lecturer browser -> PATCH .../enrollments/{student}/reject | Two attacker-controlled route ids (section, student) plus an attacker-controlled reason, driving a write to another user's enrollment. |
| reason input -> persisted rejection_reason | Free-text here would be both a data-integrity and an injection-surface problem; the enum closes it. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-05-IDOR | Elevation of Privilege | `RejectEnrollmentRequest::authorize()` and `SectionController@show` | mitigate | SEC-03 per-subject ownership: `$section->subject->lecturers()->whereKey($user->id)->exists()` — in the Form Request for the write, inline `abort_unless` for the read (no Form Request backs a GET), matching SectionController@destroy's established precedent. Falling back to D-09's `return true;` here would let any lecturer reject any student from any subject. 08-02's non-assigned-lecturer 403 test is the gate. |
| T-08-05-BIND | Tampering | Nested {section}/{student} route binding | mitigate | Route-model binding only proves both rows EXIST; it does not prove the student is enrolled in that section. An explicit integrity check aborts 404 otherwise, mirroring GradeAnswerRequest's Phase 5 precedent. |
| T-08-05-ENUM | Spoofing | `reason` input | mitigate | `Rule::enum(RejectionReason::class)` — server-validated against the fixed 5-value set; an arbitrary string is a 422. The Blade select's option list is generated from the same enum, so UI and validation cannot drift. |
| T-08-05-MA | Tampering | The `Enrollment` write (Pivot => `$guarded = []`) | mitigate | Literal, explicitly-keyed array with only `status` and `rejection_reason`; the validated array is never spread. |
| T-08-05-DOS | Denial of Service | Rejecting a student who has an in-progress attempt | mitigate | Handled upstream by 08-03's ownership-only `AttemptPolicy` (REQUIREMENTS.md Decision #7). Without it, this action would 403 the student's own in-progress attempt mid-exam and destroy unsaved work. AVL-04's regression suite is the guard; `--filter=Attempt` is re-run here. |
| T-08-05-LEAK | Information Disclosure | The roster page | mitigate | Reachable only by an assigned lecturer; students have no route to it (`role:lecturer` group + ownership check). |
| T-08-05-SC | Tampering | npm/pip/cargo installs | accept | No package-manager install occurs. |
</threat_model>
