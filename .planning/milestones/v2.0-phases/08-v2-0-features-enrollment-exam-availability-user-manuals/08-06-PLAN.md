form, and seed a demo exam that shows the window off.

null or as an empty string?).

parity pass those two forms have been missing since Phase 7, and a seeded demo window.

  <action>
    Add to `rules()` in BOTH StoreExamRequest and UpdateExamRequest, mirroring
    StoreSectionRequest's opens_at/closes_at shape but nullable rather than required:
    - `'available_from' => ['nullable', 'date']`
    - `'available_until' => ['nullable', 'date', 'after:available_from']`
    Keep the two files' rules() arrays identical to each other, as they already are. Follow
    02-06's recorded precedent: these requests deliberately duplicate rather than share a base
    class — both stay self-contained and test-covered.

    Note on `after:available_from`: when available_from is null/absent, Laravel's `after` rule
    with a missing reference field does not block the value — so a lone available_until still
    validates. 08-02's "only available_until set" test covers exactly this; if it fails, the
    correct fix is a conditional rule (apply `after:available_from` only when available_from is
    present), NOT dropping the ordering check.

    Assumption A1 (08-RESEARCH.md, LOW confidence, flagged for verification here rather than
    assumed): an unfilled `<input type="datetime-local">` submits as an empty STRING, not as an
    absent key. The research expects Laravel 11's default ConvertEmptyStringsToNull middleware to
    coerce it to null before validation runs, making `nullable|date` sufficient. Read
    bootstrap/app.php to confirm the middleware is active, then let 08-02's blank-empty-string
    test decide it. If that test fails with a 422, add a `prepareForValidation()` to BOTH requests
    that normalises empty-string availability inputs to null — a small, explicit fix. Do not
    "solve" it by making the test omit the keys; the browser will send empty strings regardless
    of what the test does. Record the outcome in the SUMMARY either way — it closes a logged
    assumption.

    Do NOT touch `UpdateExamRequest::authorize()`. Its `! $this->route('exam')->is_published`
    draft-only gate is a binding decision, not an oversight. Add a sentence to its existing doc
    comment noting that the availability window is deliberately inside that gate — set the window
    before publishing — so a future reader does not read the 403 as a bug and carve an exception.
    `ExamPublishedEditGateTest` already covers the gate and must stay green.
  </action>
