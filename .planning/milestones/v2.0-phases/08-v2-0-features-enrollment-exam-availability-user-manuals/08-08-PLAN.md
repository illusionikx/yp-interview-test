<objective>
Warn a student who tries to close the tab or navigate away mid-attempt, and make sure that
warning never fires on the two legitimate exits: the intentional submit and the timer's own
auto-submit.

Purpose: AVL-05, the last in-progress safeguard. It is purely client-side — the server cannot
detect a tab close and correctly has no role. The entire risk is the detach: a listener that
cannot be removed turns every normal submit into a confusing "are you sure you want to leave"
dialog, which is worse than having no warning at all.

Output: one Blade/Alpine change plus a human-verify checkpoint — because this behavior has no
automated vehicle in this project, and this plan says so rather than faking an assertion.
</objective>

Specifically: the page-level x-data="attemptTimer(...)" and x-init="init(); start()" at lines ~29-36; the deadline-expired listener at line 36; the submit-confirm <form method="POST"> inside the x-modal at line ~211; the attemptTimer() factory at lines ~249-336 including init() at ~270 and autoSubmit() at ~322.)
    - .planning/phases/.../08-RESEARCH.md (Pattern 6 — the exact insertion points and the browser-behavior caveats)
    - .planning/phases/.../08-UI-SPEC.md (the "Browser-native safeguard" copy row: no custom message)
    - .planning/phases/.../08-PATTERNS.md (the attemptTimer() insertion diff)
  </read_first>
  <action>
    Extend the EXISTING `attemptTimer()` Alpine factory in place — do not add a second Alpine
    component, and do not restructure the factory. 07-01's recorded lesson applies to this file:
    the whole page lives inside one attemptTimer() scope, and per-question cards have their own
    nested x-data — keep it that way.

    In `init()` (which already calls setBucket(false) and render()), create a NAMED handler stored
    on the component instance — e.g. `this._beforeUnloadHandler = (event) => { event.preventDefault();
    event.returnValue = ''; }` — and register it with
    `window.addEventListener('beforeunload', this._beforeUnloadHandler)`.
    It MUST be a stored named reference. An anonymous inline handler cannot be passed to
    removeEventListener, so it could never be detached, and every intentional submit would then
    raise the dialog — the exact failure AVL-05 defines as out of bounds.

    Attach only while the attempt is genuinely in progress. `AttemptController@show` already
    redirects away when status !== 'in_progress', so reaching this page implies in-progress — but
    assert it explicitly at the attach site (guard on the value the page already has) rather than
    relying on that upstream redirect as an invisible precondition.

    Do NOT set a custom message string. Modern Chrome, Firefox, and Safari ignore any string
    placed in `event.returnValue` or returned from the handler and show only their own generic
    dialog; Chrome removed custom-message support in Chrome 51 because sites abused it. The
    `event.returnValue = ''` line is the harmless legacy-compat idiom and is kept for that reason
    only. 08-UI-SPEC.md states this explicitly.

    Add a `detachBeforeUnload()` method calling
    `window.removeEventListener('beforeunload', this._beforeUnloadHandler)`. Make it safe to call
    twice (a no-op if already detached) — both exit paths can plausibly run in sequence.

    Wire the two intentional exits:
    - `autoSubmit()`: call `detachBeforeUnload()` BEFORE the axios POST, not after. Place it right
      after the existing `this.autoSubmitting = true;` guard line and before `this.display = '00:00'`.
      The redirect happens in the POST's `.finally()`, so detaching afterward would race the
      navigation.
    - The intentional-submit `<form method="POST">` inside the x-modal (~line 211): add
      `x-on:submit="detachBeforeUnload()"`. This is a native form POST that really navigates away;
      the submit event fires synchronously before the browser begins unloading, so the listener is
      reliably gone before beforeunload would otherwise fire.
    Note the asymmetry and why it exists: autoSubmit() is an axios call plus a manual
    `window.location.href` assignment, while the modal form is a real browser navigation. Both are
    intentional exits, both must detach, but they detach at different points in their own flow.

    Change nothing else: the countdown, the bucket/render logic, the deadline-expired event, the
    per-question autosave scopes, and FIX-01's answered-count tracking all stay exactly as they
    are.

    One browser caveat worth a comment at the handler: the dialog only fires if the page has
    received at least one user interaction ("sticky activation"). That is a browser anti-abuse
    measure the app cannot control — during manual verification, click somewhere on the page
    first. A no-interaction page not showing the dialog is correct browser behavior, not a bug.
  </action>

    <automated>php artisan test --filter=AttemptShowTest</automated>
