    Create app/Http/Controllers/Student/EnrollmentController.php with store() and destroy().

    store(EnrollRequest $request, Section $section):
    1. Window check first (cheap rejection, no lock needed). Using the half-open rule: if
       now < opens_at, redirect back with the red flash
       "Enrollment for this section hasn't opened yet."; if now >= closes_at, redirect back with
       "Enrollment for this section is closed." Reuse `$section->windowStatus()` from 08-01
       rather than re-deriving the comparison — one implementation, three consumers.
    2. Then the locked transaction — the ENR-02 mechanism. Inside `DB::transaction()`:
       re-read the section with `Section::whereKey($section->id)->lockForUpdate()->first()`.
       This exclusive row lock is what serialises concurrent applies to the SAME section: a
       second request blocks at ->first() until the first transaction commits, so the count it
       then reads is guaranteed current. Count enrolled students live via
       `$locked->enrollments()->wherePivot('status', EnrollmentStatus::Enrolled->value)->count()`
       — do NOT add a denormalized counter column (CLAUDE.md: no denormalization without a
       measured problem; the live count inside the lock is the correct and simplest answer).
       If the count >= $locked->capacity, refuse.
       Then the ENR-04 guard, inside the SAME transaction so both checks are consistent under
       one lock: does this student hold another active enrollment in a DIFFERENT section sharing
       this section's subject_id + year + semester? Query Enrollment with
       `whereHas('section', ...)` (this is why 08-01 added the section() relation) and
       `lockForUpdate()`. If so, refuse.
       Otherwise write with
       `Enrollment::updateOrCreate(['section_id' => ..., 'user_id' => ...], ['status' => EnrollmentStatus::Enrolled, 'rejection_reason' => null])`.
       updateOrCreate — never create() — because the unique(section_id, user_id) row already
       exists for anyone who previously withdrew or was rejected; create() would throw a
       duplicate-key QueryException (MySQL 1062). Set rejection_reason explicitly to null (not
       '') so a stale reason can never linger on a fresh enrolled row.
    3. Signal refusals OUT of the closure by returning a sentinel value from
       `DB::transaction(...)` (it returns the closure's return value) and mapping it to flash
       copy in the controller body. Do NOT abort(409) and do NOT introduce dedicated exception
       classes: 08-UI-SPEC.md requires these refusals to appear as red flash banners on the
       sections page, and an abort renders an error page instead. A returned sentinel keeps this
       to one small method with no new files.
    4. On success, redirect back with the green flash
       "You're enrolled in {section name}." using `$section->name` (the computed accessor).

    Both write arrays must be LITERAL and explicitly keyed. `user_id` always comes from
    `$request->user()->id`, never from input. Enrollment extends Pivot, whose `$guarded = []`
    makes every column — including `status` and `rejection_reason` — mass-assignable; forwarding
    `$request->all()` here would let a student POST their own `status` or someone else's
    `user_id`. 08-02's tests assert both forgeries fail.

    destroy(Section $section): ENR-03. Refuse when now >= closes_at with the red flash
    "You can no longer withdraw — this section's enrollment window has closed." (half-open: AT
    the boundary is already closed). Otherwise update the caller's own enrollment row —
    scoped to `auth()->id()`, never a request-supplied user_id — to status Withdrawn, and
    redirect back with "You've withdrawn from {section name}." No lock is needed: this is a
    single-row write by its owner with no shared invariant.

    Add the routes to routes/student.php inside the existing
    `['auth', 'verified', 'role:student']` group, matching its flat style. Use exactly the names
    08-02 pinned: `student.sections.enroll` (POST sections/{section}/enroll) and
    `student.sections.withdraw` (DELETE sections/{section}/enroll).

    Do NOT add an observer or model event on Enrollment for status transitions — CLAUDE.md
    forbids hidden-side-effect model hooks (the grading-via-observer prohibition), and the same
    discipline applies here: every transition is an explicit controller call.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| student browser -> POST /student/sections/{section}/enroll | Untrusted input drives a write to `enrollments`, a Pivot model on which every column is mass-assignable. |
| student browser -> DELETE /student/sections/{section}/enroll | Untrusted input drives a status write; the target row must be the caller's own. |
| concurrent student requests -> sections.capacity invariant | The one genuine multi-user race in this phase; the invariant lives in the DB, not in PHP. |
| controller -> view | The sections page must expose aggregate counts only, never other students' identities. |
