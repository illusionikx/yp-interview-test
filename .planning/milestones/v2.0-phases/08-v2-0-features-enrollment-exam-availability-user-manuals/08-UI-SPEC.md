# Phase 8 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Inheritance:** This phase builds entirely on the **LOCKED** design system shipped in `07-UI-SPEC.md` — Flowbite top-navbar shell (no sidebar), Flowbite-default-blue accent, the 8-pt spacing scale, the 4-size/2-weight type scale, class-based dark mode, comfortable density, and the semantic status-pill palette (`x-status-pill`). None of that is re-decided here. This contract specifies only what's genuinely **new** in Phase 8: the enrollment browse/apply/withdraw surfaces, the lecturer section-roster/reject surface, the exam-availability form fields and pre-start page, and the in-app help manuals.

Inherited verbatim from `07-UI-SPEC.md` — no new tokens, no new exceptions.

| lg | 24px | Card internal padding (desktop) |
| xl | 32px | Layout gaps between cards/sections |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level top/bottom spacing |

Exceptions: none new. The 44px touch-target exception (icon-only controls) still applies to the dark-mode toggle and hamburger; no new icon-only controls are introduced this phase (Reject/Withdraw/Apply are all labeled text buttons).

Phase-8-specific application:
- The manuals (DEL-04/DEL-05) are long-form text content: use `lg` (24px) vertical rhythm between paragraphs/list-groups within a manual section, and `2xl` (48px) between top-level manual sections (e.g. between "Enrolling in a Section" and "Taking a Timed Exam").

## Typography

Inherited verbatim from `07-UI-SPEC.md` — exactly 4 sizes, exactly 2 weights (400 regular, 600 semibold). No new roles added.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px (`text-base`) | 400 regular | 1.5 |
| Label | 14px (`text-sm`) | 600 semibold | 1.4 — table headers, form labels, status pills |
| Heading | 20px (`text-xl`) | 600 semibold | 1.2 — card/panel titles, modal titles, manual section headings |
| Display | 28-30px (`text-3xl`) | 600 semibold | 1.2 — page-level `$header` slot title |

Manual pages (long-form reading, new this phase):
- Body role (16px/400/1.5) for all manual paragraph and list-item text — do not introduce a "prose" size variant; the manuals are short task-based walkthroughs, not dense documentation, so Body at 1.5 line-height is sufficient for readability.
- Heading role (20px/600) for each manual task section (e.g. "Enrolling in a Section", "Withdrawing", "Taking a Timed Exam") — one `<h3>` per task, matching the number of success-criteria flows.
- Numbered steps within a task use a plain `<ol class="list-decimal list-inside space-y-2">` at Body size — no custom counter styling.
- Container width: `max-w-3xl` (matches `student.exams.show`'s existing reading-width precedent) — narrower than the `max-w-7xl` table pages, because this is prose, not tabular data.

Inherited verbatim from `07-UI-SPEC.md` — Flowbite default blue accent, no custom Tailwind color config. Phase 8 only **extends the reserved-for lists** with the specific new elements it introduces; it does not add new colors.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `bg-gray-50` light / `dark:bg-gray-900` dark | Page background |
| Secondary (30%) | `bg-white` light / `dark:bg-gray-800` dark | Cards, tables, modals, navbar |
| Accent (10%) | `blue-600` light / `blue-500` dark | See extended list below |
| Destructive | `red-600` light / `red-500` dark | Destructive actions only, see extended list below |

Accent (`blue-600`/`dark:blue-500`) reserved for — Phase 7 list **plus**:
- "Apply" button (student section-apply)
- "Proceed" button (pre-start page)
- Text links: "Withdraw" trigger (opens the confirm modal — the trigger itself is a link/button in accent color; the modal's confirm action is destructive red, see below), "View roster" (lecturer section link), "Help" nav item active/hover state
- Focus rings on all new form controls (`available_from`/`available_until` datetime inputs, the rejection-reason `<select>`)

Destructive (`red-600`/`dark:red-500`) reserved for — Phase 7 list **plus**:
- "Withdraw" confirm button inside the withdraw confirmation modal (the trigger link that opens the modal stays accent-colored per Flowbite convention already used for "Delete" triggers; only the modal's final destructive action button is red)
- "Reject Student" confirm button inside the reject confirmation modal

### Status Pill Palette — extending the locked semantic system

The four palettes and their existing `x-status-pill` match arms (`enrolled`/`published`/`open` → green, `rejected`/`closed` → red, `full` → amber, default → gray) are **locked and unchanged**. Phase 8 introduces new status keywords that must map onto these same four palettes. To keep the mapping explicit rather than relying on the gray default as an accident of omission, **extend `x-status-pill`'s `match()` with these additional explicit arms**:

| New status keyword | Palette | Applies to |
|---------------------|---------|-----------|
| `'available'` | Green (alias of `open`) | Exam availability list — "Available" |
| `'withdrawn'` | Gray (explicit, currently falls through `default` — make it an explicit arm for clarity) | Student's own withdrawn enrollment state |
| `'opening'` | Gray (explicit — used for exam "Opens {date}" and section "Opens {date}", both dynamic-label states) | Exam not-yet-available, section not-yet-open |

No new arms needed for `rejected`, `closed`, `full`, `enrolled`, `published`, `open` — those already exist and are reused as-is (e.g. exam "Closed" reuses the existing `closed` → red arm; section "Open" reuses the existing `open` → green arm).

The pill's **slot text is always the caller-supplied label** (e.g. `Opens Aug 12`, `28/30` is NOT a pill — see below), independent of the `status` prop used for color selection.

Capacity (`28/30`) is **not** a pill — it is plain Label-role text (14px/600) directly beside the FULL pill when at capacity, per 08-CONTEXT.md ("Capacity rendered as text `28/30` plus the amber FULL pill at capacity").

## Phase Notes (surface-specific, for planner/executor)

### 1. Student enrollment browse (ENR-01, ENR-02, ENR-05, ENR-06)

- **IA:** subject list (`student.subjects.index` — Claude's discretion on exact route name) → that subject's sections page (`student.subjects.sections.index`-shaped). Mirrors the lecturer's existing subject→section nesting pattern (`lecturer.subjects.sections.*`).
- **Layout:** reuse the existing table pattern from `lecturer/sections/index.blade.php` (grouped-by-subject card + table). For the student's single-subject sections page: one card, one table. Columns: Section | Capacity | Enrollment Window | Your Status / Action.
- **Capacity cell:** plain text `28/30` (Label role, 14px/600); when at capacity, append the amber FULL pill immediately after the text (`28/30` + `<x-status-pill status="full">FULL</x-status-pill>`).
- **Window-status cell:** reuse the exact `$windowStatus`/`$windowLabel` computed-in-view pattern already established in `lecturer/sections/index.blade.php` (lines 46-56) — `opens` (gray, dynamic date label), `closed` (red), `open` (green) — apply the same `now()->lt(opens_at)` / `now()->gte(closes_at)` half-open-interval logic per 08-CONTEXT.md Decision 6 (`[opens_at, closes_at)`).
- **Action/status cell** — one of four mutually exclusive states per section, per student:
  - No enrollment + window open + not full → "Apply" button (accent, direct POST, no modal, per locked decision — immediate enroll).
  - No enrollment + window not open, or closed, or full → no button, window-status pill from the cell above is the only indicator (FULL pill or Opens/Closed pill already communicates why).
  - Active enrollment on this section → green **Enrolled** pill + "Withdraw" text link (triggers the confirm modal).
  - Rejected from this section → red **Rejected** pill + "Rejected: {reason}" inline text (Body role, `text-gray-600 dark:text-gray-400`) + "Apply" button again if the window is still open (re-apply per ENR-05).
  - Withdrawn from this section → gray **Withdrawn** pill + "Apply" button again if the window is still open (re-apply per ENR-05).
- **One active enrollment per subject/semester (ENR-04):** if the student already holds an active enrollment in a different section of the same subject+semester term, that other section's row shows no Apply button even if otherwise eligible — render a plain gray note instead of a button: "Enrolled in another section this semester" (Body role, muted). This is a display consequence of the server-side one-active-enrollment rule, not a new pill state.

### 2. Withdraw & Rejection

- Withdraw modal follows the exact `x-modal` pattern already used for "Delete Section" (see `lecturer/sections/index.blade.php` lines 72-85) — same modal chrome, same `x-secondary-button`/`x-danger-button` pairing, different copy (see Copywriting Contract above).
- **Lecturer section roster page** (`lecturer.sections.show`, new): table of enrolled students. Columns: Student | Enrolled Since | Action. "Reject" button per row opens the reject modal (per-row, mirroring the per-section delete-modal-per-row pattern already established). The modal's reason `<select>` uses the same `x-input-label`/`<select>` styling already used for `semester` in `lecturer/sections/create.blade.php`.
- Roster page card/table styling matches `lecturer/subjects/index.blade.php`'s existing table conventions (header row `text-sm font-semibold text-gray-500 dark:text-gray-400`, body rows `divide-y divide-gray-200 dark:divide-gray-700`).

### 3. Exam availability & pre-start (AVL-01, AVL-02, AVL-03, AVL-04, AVL-05)

- **Form fields:** add `available_from`/`available_until` as a `grid grid-cols-2 gap-4` pair to the existing exam create/edit form (`lecturer/exams/create.blade.php`, `edit.blade.php`), directly following the existing `duration_minutes` field. Use the exact `type="datetime-local"` `x-text-input` pattern already established for `opens_at`/`closes_at` in `lecturer/sections/create.blade.php` — but **not** `required` (both fields optional, empty = unbounded on that side, per 08-CONTEXT.md).
- **Pre-start page:** enhance `student/exams/show.blade.php` in place (same route, same card layout, `max-w-3xl` container already there) — add: instructions/description (already rendered), the availability window (new — render as two lines or a single "Available: {from} – {until}" line, omitting either side if unbounded, using the `available`/`opening`/`closed` pill next to the exam title per the extended status-pill table above), and "your enrolled section" (new — the section name the student is enrolled in for this exam's subject). Replace the existing button copy ("Start Exam"/"Resume Exam") with "Proceed" per the locked decision, and add a "Back" secondary link (`x-secondary-button`-styled or plain underlined text link, matching the existing "Back to my exams" link already at the bottom of the page) that returns to `student.exams.index`.
- **Student exam list** (`student/exams/index.blade.php`): add the availability pill (green Available / gray Opens {date} / red Closed) beside each exam title, using the extended `available`/`opening`/`closed` status keywords.
- **In-progress safeguard (AVL-05):** implement as a small Alpine/vanilla-JS `beforeunload` listener scoped to the attempt-taking page only (`student/attempts/show.blade.php`), attached on page load while `status === 'in_progress'`, explicitly removed (`window.removeEventListener`) immediately before the submit form's actual submission and before the auto-submit timer's own submit call — no visual UI element, no custom modal (per Copywriting Contract above).

### 4. In-app help manuals (DEL-04, DEL-05) — user override, in-app not Markdown

- Two Blade pages inside the standard `x-app-layout` shell: one student manual, one lecturer manual (role-scoped route, e.g. `student.help.show` / `lecturer.help.show` — exact naming at Claude's discretion).
- **Layout:** `max-w-3xl` centered container (reading width, not the `max-w-7xl` table width), `$header` slot = "Student Manual" / "Lecturer Manual" (Display role), one card (`bg-white dark:bg-gray-800 rounded-lg p-6`) containing all task sections in a single scroll — no tabs/accordions/JS interactivity needed for a short manual.
- **Content structure per manual** (task-based, mirrors the phase's success-criteria flows — not a feature reference list):
  - Student manual: "Enrolling in a Section", "Withdrawing", "Checking Exam Availability", "Taking a Timed Exam", "Viewing Your Results" — five `<h3>` (Heading role) sections, each a short paragraph + numbered steps referring to on-screen labels by their exact visible text (e.g. "Click **Apply** next to the section you want").
  - Lecturer manual: "Managing Subjects and Sections", "Viewing a Roster and Rejecting a Student", "Authoring and Assigning an Exam with an Availability Window", "Grading" — four `<h3>` sections, same structure.
- **Text-only, no screenshots** (locked decision — they rot and add no value in a graded repo). Refer to UI elements by their exact visible label/copy as defined in this contract and in `07-UI-SPEC.md`, so the manual and the shipped UI never drift apart on wording.
- **Nav placement:** append a single "Help" link (accent-colored on hover/active, same styling as the existing role-scoped nav links in `navigation.blade.php`) after the last role-scoped nav item — after "Exams" for lecturer, after "My Exams" for student (with "Enroll" inserted before "My Exams" per the deferred-from-Phase-7 nav item). Final nav order: Lecturer → Subjects / Sections / Exams / Help. Student → Enroll / My Exams / Help. Applies to both the desktop nav row and the responsive mobile menu.
- **Written last** in the phase's plan sequence, after all other Phase 8 UI has shipped, so the copy accurately reflects the real screens (per STATE.md sequencing guidance and 08-CONTEXT.md).
