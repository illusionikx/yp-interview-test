<objective>
Fix the latent authorization bug that becomes load-bearing the moment withdraw/reject ship:
`AttemptPolicy::view()` and `update()` currently derive access from `Exam::visibleTo()`, which
is enrollment-status-driven. Change both to ownership-only, and pin the behavior with the
AVL-04 critical regression suite.

Purpose: this is the single highest-risk change in Phase 8 and it must land BEFORE the code
paths that can mutate a student's enrollment status exist (08-04's withdraw, 08-05's reject).
Today nothing exercises the failure path — every existing fixture keeps students Enrolled for
the duration of an attempt — so the bug is invisible. From 08-04 onward, a withdrawn or
rejected student's own in-progress attempt would start 403'ing on every autosave and on submit,
discarding their unsaved work and silently breaking AVL-04, TAK-03 and TAK-04 at once.

This is the direct implementation of REQUIREMENTS.md Resolved Decision #7: post-start attempt
access is decoupled from live enrollment — attempts are ownership-gated, not visibility-gated;
the enrollment/availability check applies at start only.

Output: a two-line policy change and the regression suite that makes it permanent.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| authenticated student -> another student's attempt | The IDOR boundary. This plan relaxes one condition of the access check and must not relax this one. |
| enrollment state (mutable from 08-04/08-05) -> in-progress attempt access | The coupling being removed. Mutable state on one side of this boundary must not retroactively revoke access on the other. |
| exam publication/assignment state -> in-progress attempt access | Same class of coupling, same fix. |
