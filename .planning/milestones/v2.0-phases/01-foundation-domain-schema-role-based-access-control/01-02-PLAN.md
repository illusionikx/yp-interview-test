| request input → User model | `role`/`classroom_id` become mass-assignable here; any controller using request data to build a User could over-set them |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-02-MA | Tampering / Elevation of Privilege | `User::$fillable` now includes `role`/`classroom_id` | mitigate | Fillable membership is required for server-controlled writes (seeder, factories), but a documented invariant is added: no public-facing controller may build a User from `$request->all()`/request-sourced `role`. Registration (01-04) sets `role` as a server constant; there is no other public write path this phase. Reviewed against RESEARCH §Security Domain mass-assignment pattern. |
| T-01-01-CAST | Tampering | Raw `role` string bypassing the enum | mitigate | `casts()` maps `role => Role::class` so every read/write is enum-checked; helpers `isLecturer()`/`isStudent()` are the only sanctioned role checks (no scattered raw string comparisons — RESEARCH Anti-Pattern) |
| T-01-SC | Tampering | Supply chain | accept | No new Composer packages this phase (01-RESEARCH.md Package Legitimacy Audit) |
