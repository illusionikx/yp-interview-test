<objective>
Build the **RBAC walking-skeleton loop** on top of the enum/model layer: a parameterized `EnsureUserHasRole` middleware aliased as `role` in `bootstrap/app.php` (D-07), two role-gated route areas (`routes/lecturer.php`, `routes/student.php` — D-06) each landing on a minimal placeholder view, and a `DashboardController` that dispatches the shared Breeze `/dashboard` route to the correct area by role (D-08).

This is the vertical slice recorded in SKELETON.md: after this plan a Student who guesses a `/lecturer/*` URL is blocked server-side (403 — RBAC-04) and a logged-in user lands in their own area (RBAC-03). The block is middleware-enforced at the group level, never "hide the nav link" (RESEARCH Anti-Pattern 2; Security Domain forgotten-middleware threat).

Purpose: RBAC-03 (role-based post-login redirect) and RBAC-04 (server-side role gating).
Output: middleware + alias, two route-group files, two placeholder views, DashboardController, and three tests (unit middleware gate + feature gating + feature redirect).
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → /lecturer/* and /student/* | Untrusted URL access; a wrong-role (or guest) user must be denied before any controller code runs |
| browser → /dashboard | Shared post-login entry; must fan out to the correct area without leaking the other |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-03-MW | Elevation of Privilege | A future lecturer/student route added without a role gate | mitigate | Role middleware declared once at the GROUP level in `routes/lecturer.php`/`routes/student.php` (D-06/D-07); any route added inside the group inherits `role:*` automatically — proven by RoleMiddlewareTest |
| T-01-04-HIDE | Elevation of Privilege | "Security through hidden nav link" | mitigate | Enforcement is server-side `abort(403)` in `EnsureUserHasRole`, unit-tested in Task 1 and feature-tested in Task 3 — never Blade `@if` alone (RESEARCH Anti-Pattern 2) |
| T-01-05-VERIF | Spoofing / Access Control | Unverified account reaching a role area | mitigate | Route groups keep Breeze's `verified` middleware (RESEARCH A3); seeded test accounts set email_verified_at in 01-04 so verification does not silently break the flow (Pitfall 1) |
| T-01-SC | Tampering | Supply chain | accept | No new Composer packages this phase (01-RESEARCH.md Package Legitimacy Audit) |
