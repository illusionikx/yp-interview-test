<objective>
Close the two remaining halves of the RBAC walking-skeleton loop's "entry" side: (1) lock public registration to the Student role so there is no public path to a privileged Lecturer account (RBAC-02 / D-09), and (2) seed the minimal verified Lecturer + Student pair that makes the role gating and redirect (success criteria #3/#4) actually verifiable this phase (D-10).

Both are pure additions to untouched Breeze scaffold: `RegisteredUserController@store` currently does a plain `User::create` with no `role` field, and `DatabaseSeeder` currently creates one generic test user. The critical correctness details: the role is a **server constant, never sourced from request input** (the self-elevation threat RBAC-02 exists to close), and the seeded accounts **explicitly set `email_verified_at`** or Breeze's `verified` middleware silently blocks them (Pitfall 1).

Purpose: RBAC-02 — public self-registration always creates a Student; Lecturers are seeded.
Output: hardened `RegisteredUserController@store` + a new RegistrationTest case, and a minimal idempotent `DatabaseSeeder` + `TestAccountSeederTest`.
</objective>
