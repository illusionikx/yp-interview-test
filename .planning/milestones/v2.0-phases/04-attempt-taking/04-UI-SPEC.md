# Phase 4 — UI Design Contract

> Visual and interaction contract for the student take-exam flow. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> This is the one truly interactive, student-facing screen in the app, and the phase where a client-trusted timer or a leaked `is_correct` would be a correctness/security failure, not just a cosmetic bug. Every visual rule below is subordinate to the locked backend contract in `04-CONTEXT.md` (D-01..D-10) and `04-RESEARCH.md`: **the countdown is display-only, the server is the sole timing authority, and option markup never carries `is_correct`.**

## Design System

| Property | Value |
|----------|-------|
| Tool | none — Breeze Blade + Tailwind 3 + Alpine.js (fixed stack per PROJECT.md; not a React/Next.js/Vite-framework project, so the shadcn gate does not apply) |

Where a glyph is needed (checkmark on the confirmation page, a small "saved" tick), use a single inline `<svg>` (Heroicons-style outline path, hand-copied, no new dependency) or plain Unicode/text. Never `npm install` an icon package for this phase. |
| Font | Figtree (already loaded via `layouts/app.blade.php` `<link>` + `fontFamily.sans` in `tailwind.config.js`) — no change |

**Codebase-grounded palette already in use (from `resources/views/components/*` and `student/exams/show.blade.php`):**
- Page background: `bg-gray-100`
- Card/panel surface: `bg-white`, `shadow-sm`, `rounded-lg`
- Primary button: `bg-gray-800` / `hover:bg-gray-700` (NOT indigo — this is the existing house style for primary CTAs; the Submit button reuses it, does not introduce a new color)
- Secondary button: `bg-white border-gray-300`
- Destructive button: `bg-red-600 hover:bg-red-500`
- Focus/accent: `indigo-500`/`indigo-600` (focus rings, links, `@tailwindcss/forms` native radio/checkbox accent color)
- Meta/muted text: `text-gray-500`, `text-sm`

This phase does not introduce a new palette — it extends the existing one with the minimum new semantic states (timer escalation, save status) documented below.

## Spacing Scale

Declared values (must be multiples of 4) — matches the scale already implicit in the reused Breeze components, no new tokens introduced:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Gap between a radio input and its option label; gap inside the autosave status tag |
| sm | 8px | Gap between stacked form controls; gap between the point-value badge and question body |
| md | 16px | Padding inside a question card (`p-4`); gap between question cards on mobile |
| lg | 24px | Card padding on the take page (`p-6`, matches existing `show.blade.php`); gap between question cards on desktop |
| xl | 32px | Vertical rhythm between the sticky timer header and the first question |
| 2xl | 48px | Space above/below the Submit section at the bottom of the question list |
| 3xl | 64px | Vertical centering offset on the Submitted confirmation page |

Exceptions:
- **44px minimum touch target** on the Submit button and the confirm-modal's two action buttons (`py-3 px-4` minimum on mobile breakpoints) — this is a timed, high-stakes action; a mis-tap must not be easy.
- The sticky countdown header ignores the page's normal `max-w-3xl` card padding and spans full viewport width with its own `px-4 sm:px-6 lg:px-8` — it must stay visible above the fold at all scroll positions, so its internal spacing is independent of the question-card rhythm below it.

|------|-------|-------|

### Status Colors (phase-specific extension — countdown & autosave only)

The base 60/30/10 table above doesn't cover the two stateful indicators this phase requires (countdown escalation, autosave status). These are semantic status colors, not decorative accents, and are scoped exclusively to the two components named:

| State | Color | Applies to |
|-------|-------|------------|
| Countdown — normal | `text-gray-800` on `bg-indigo-50` badge | `remaining_seconds > 300` (more than 5 minutes left) |
| Countdown — warning | `text-amber-700` (`#B45309`) on `bg-amber-50` badge, `border-amber-300` | `60 < remaining_seconds <= 300` (5 minutes down to 1 minute) |
| Countdown — critical | `text-red-700` on `bg-red-50` badge, `border-red-300`, `animate-pulse` (core Tailwind utility) | `remaining_seconds <= 60` (last minute) |
| Autosave — saving | `text-gray-500`, no icon, label "Saving…" | Request in flight |
| Autosave — saved | `text-green-700` (`#15803D`), inline checkmark glyph, label "Saved" | 200 response received |
| Autosave — failed | `text-red-600`, label "Save failed — Retry" rendered as an underlined `<button>` | Non-2xx response (network error or, notably, a 422 deadline-rejected write — see Interaction Contract) |

Thresholds (300s / 60s) are fixed, absolute values applied regardless of the exam's total `duration_minutes` — simplest correct rule per Claude's Discretion in `04-CONTEXT.md`; a sub-5-minute exam simply starts in the warning state, which is accurate, not a bug.

## Screen Inventory & States

Four screens/states cover this phase's entire surface. No separate "read-only expired" view exists by design (see Interaction Contract) — an expired attempt always redirects straight to the Submitted Confirmation screen rather than rendering a disabled question form, which removes an entire class of stale-form bugs.

### 1. Take-Exam Page (`in_progress` attempt)
- **Layout:** single column, `max-w-3xl mx-auto`, matches the existing `student/exams/show.blade.php` container width.
- **Sticky header** (`sticky top-0 z-10 bg-white shadow`): exam title (Heading token, truncated with `truncate` if long) on the left; countdown badge (Display token, state-colored per Status Colors) on the right. On mobile, stack title above countdown if width is insufficient — never let the countdown scroll out of view.
- **Question list:** one card per question (`bg-white rounded-lg shadow-sm p-6`, `space-y-6` between cards). Each card:
  - Meta line (Label token, `text-gray-500`): "Question {n} of {total} · {points} {points === 1 ? 'point' : 'points'}"
  - Question body (Body token, `font-semibold` — an explicit exception to the Body row's 400 weight, matching the existing question-body emphasis pattern from Phase 2's `_form.blade.php`)
  - MCQ: `<fieldset>` with a visually-hidden (`sr-only`) `<legend>` duplicating the question body for screen readers; one `<label>` per option wrapping a native `<input type="radio" name="question_{id}">` + the option body text. Options render **body text only** — see Interaction Contract's hard rule.
  - Open-text: `<label>` (visually can reuse the question body itself as the label via `for`/`id` association) + `<textarea rows="4">`, reusing `x-text-input`'s border/focus styling conventions.
  - Per-question autosave tag (Label token, right-aligned under the input) in one of the three Status Colors states.
- **Submit section** (bottom of the list, `mt-12 flex justify-end`): the Submit Exam button (see Copywriting).

### 2. Confirm-Submit Modal
- Reuses the existing `x-modal` component verbatim (no new modal markup) — `maxWidth="sm"`.
- Content: heading, body with answered/total count, Cancel + Confirm buttons per Copywriting Contract.
- Keyboard/focus: `x-modal` already handles focus trap, `Escape` to close, and `Tab`/`Shift+Tab` cycling — no additional work required, just slot the content in.

### 3. Auto-Submit Transition (client-side, transient)
- Triggered when the Alpine countdown reaches 0, OR when any in-flight autosave/submit request comes back 422 because the server-side deadline was crossed mid-request.
- Full-width banner (`bg-gray-800 text-white`, NOT a modal — no dismiss action exists, so it must not trap focus) pinned above the sticky header: "Time's up — submitting your exam…"
- All form controls become `disabled` immediately (prevents further edits while the redirect is in flight).
- Duration: as long as the auto-submit POST takes, typically well under 1s; the page then hard-redirects to Screen 4.

### 4. Submitted Confirmation Page
- **Layout:** centered card (`max-w-md mx-auto`, `3xl` token vertical offset from the top), no sticky header, no countdown, no question content rendered — a calm, deliberately minimal screen.
- Inline SVG checkmark (`text-green-600`, ~48px) above the heading.
- Heading, body, and "Back to my exams" link per Copywriting Contract.
- **Hard rule:** this page must not render or fetch anything score/grade-related — no "Your score" placeholder, no "pending grading" progress bar. Just the confirmation. Phase 5 owns everything past this point.

## Interaction Contract

These rules translate the D-01..D-10 backend contract into concrete UI behavior. They are non-negotiable for this phase — a checker/auditor should treat any deviation as a FAIL, not a style nit.

1. **The countdown is cosmetic.** It is seeded once from server-passed `remaining_seconds` at page load and ticks down client-side via `setInterval`. It never re-fetches the true remaining time from the server mid-session. Its only two side effects are (a) recoloring per the Status Colors thresholds and (b) firing the auto-submit POST at 0. It must never be used to gate whether a form control is enabled/disabled — that gate is the server's 422 response, surfaced via the autosave/save-failed tag.
2. **Options render body text only.** No option's markup, any Alpine `x-data` JSON payload, or any data attribute may contain `is_correct`, a boolean styled as "correct," or any styling (color, icon, bold) that would let a student infer which option is correct before grading. A checker should `grep` the rendered HTML for the literal substring `is_correct` and expect zero matches (per `04-RESEARCH.md` Pitfall 3).
3. **Autosave fires on `@change` for radios, `@blur` for textareas** (matches the debounce-free per-answer POST pattern in `04-RESEARCH.md` Pattern 4) — not on every keystroke, to avoid hammering the autosave endpoint on open-text answers.
4. **A failed autosave never blocks continued answering of other questions.** The failure is scoped to that one question's tag; the student can keep going and retry later.
5. **The Submit button is always enabled**, even with zero answers filled in — this phase does not implement a "you must answer all questions" gate (not in TAK-01..06); the confirm-modal's answered-count is informational only.
6. **No separate read-only/expired take-page view.** If `show()` is loaded (fresh visit or reload) after the deadline has passed, the server's lazy `finalizeIfExpired()` (D-05) has already flipped the attempt to `submitted` before the view renders, and the controller redirects straight to Screen 4. The take page therefore never needs a "disabled form" visual state — design and implementation effort stays on the four screens above only.
7. **Accessibility baseline:** every input has a programmatically-associated `<label>` (or MCQ `<fieldset>/<legend>`); focus is visible on every interactive element (inherited from Breeze's default focus-ring styling — do not `outline-none` without a replacement ring); the per-question autosave tag's container carries `aria-live="polite"` so a screen-reader user hears "Saved"/"Saving…" without needing to re-focus it; the countdown badge does **not** carry `aria-live` on every tick (that would spam the AT user every second) — instead, wrap only the *state transition* (normal→warning, warning→critical) in a one-shot `aria-live="assertive"` announcement, e.g. via an Alpine `$watch` on the computed state that briefly writes to a visually-hidden live region only when the bucket changes.
