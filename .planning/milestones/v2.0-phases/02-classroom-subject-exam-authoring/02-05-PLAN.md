## Phase Goal (User Story)

**As a** lecturer, **I want to** build the full teaching content model — classrooms, subjects, class-subject links, student rosters, and complete exams — **so that** everything is authored and correct before any of it is exposed to a student.

*(Derived from the Phase 2 `**Goal:**` line in `.planning/ROADMAP.md`.)*

**This slice:** *As a lecturer, I want to add multiple-choice and open-text questions (with points) to a draft exam, so that the exam has content.*

<objective>
Deliver the phase's core non-trivial capability (EXM-02, EXM-03, EXM-04; D-07, D-08): add questions to a draft exam from the exam's show page. MCQ questions use an Alpine dynamic option-row list with a single shared `correct_option` radio group (Pattern 1) so "at most one correct" is browser-native; the server re-verifies "exactly one correct + ≥2 non-blank options" in a Form Request `after()` hook. Open-text questions skip options entirely. Points default to 1 with a positive-integer rule. Persistence writes the Question plus (for MCQ) its Options inside a single `DB::transaction`, marking `is_correct` true only at the chosen index.

This slice covers question CREATE only. Question edit/delete and the question-level published-edit gate land in 02-06.

Purpose: An exam needs questions before it is worth publishing/assigning; this is the authoring heart of the phase.
Output: `Lecturer\ExamQuestionController` (store), `StoreQuestionRequest` (rules + after() hook + draft-only authorize gate), the Alpine `_form` partial, the add-question section wired into `exams/show`, and two feature tests.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser (Alpine form) → question store | The client renders/removes option rows; the posted `options[]`/`correct_option` array is untrusted and crosses into `StoreQuestionRequest`. |
| draft-state boundary | Questions may only be added while the parent exam is a draft. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-RBAC | Elevation of Privilege | questions.store route | mitigate | Inside the unchanged Phase-1 `role:lecturer` group; the tests assert a student is 403. |
| T-02-PUB | Tampering | StoreQuestionRequest.authorize() | mitigate | `authorize()` returns `! route('exam')->is_published` — a question cannot be added to a published exam even via a forged/replayed POST (Pitfall 3); the hidden add-question form on published exams is UX only. |
| T-02-MCQ | Tampering | StoreQuestionRequest.after() + controller persistence | mitigate | `after()` rejects a missing/out-of-range `correct_option`; `min:2` rejects under-count; `options.*.body required_with` rejects blank rows (Pitfall 5); `is_correct` is derived server-side from `correct_option`, never accepted from the client — a malformed MCQ that would confuse Phase-5 grading cannot be persisted. |
| T-02-MA | Tampering | question/option persistence | mitigate | Only `type`/`body`/`points` (question) and server-derived `body`/`is_correct`/`position` (options) are written; no `exam_id`/`is_correct` is taken from raw request input (the parent exam comes from the route binding). |
| T-02-SC | Tampering | package installs | N/A | No packages installed this phase (`02-RESEARCH.md`). |
</threat_model>
