## Phase Goal (User Story)

**As a** lecturer, **I want to** build the full teaching content model — classrooms, subjects, class-subject links, student rosters, and complete exams — **so that** everything is authored and correct before any of it is exposed to a student.

*(Derived from the Phase 2 `**Goal:**` line in `.planning/ROADMAP.md`.)*

**This slice:** *As a lecturer, I want to create classrooms, link subjects to them, and assign students, so that a class group exists to receive exams later.*

<objective>
Deliver the full classroom management surface: classroom CRUD (CLS-01, D-02), classroom↔subject linkage via the `classroom_subject` pivot managed from the classroom edit page (CLS-03, D-03), and student roster assign/unassign that writes `users.classroom_id` directly (CLS-04, D-04). Each capability is a testable vertical slice landed as its own task with its own named feature test.

Purpose: A classroom with its linked subjects and assigned students is the class-group entity Phase 3 will assign exams to.
Output: `Lecturer\ClassroomController`, `Lecturer\ClassroomRosterController`, three Form Requests, classroom Blade views (edit page hosts both the subject multi-select and the roster panel), routes, and three feature tests.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → classroom routes | Form input (name, subject_ids[], student_id) crosses into controllers behind Phase-1 `auth`+`verified`+`role:lecturer`. |
| roster action → users table | The one request path allowed to write another user's `classroom_id`. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-RBAC | Elevation of Privilege | classrooms.* + roster routes | mitigate | Inside the unchanged Phase-1 `role:lecturer` group; tests assert student 403 on classroom + roster routes. |
| T-02-MA | Tampering / EoP | Classroom Form Requests, roster write | mitigate | Classroom controller mass-assigns only `name`; `classroom_id` is written ONLY by `ClassroomRosterController` from a validated, student-constrained target — never via generic `User::update($request->all())` (T-01-02-MA). |
| T-02-ROSTER | Tampering / EoP | ClassroomRosterController store/destroy | mitigate | `AssignStudentRequest` constrains the target to `role=student` (a lecturer account can't be moved); `destroy` guards `$student->classroom_id === $classroom->id` with `abort_unless(...,404)` to block cross-student unassign via a mismatched route pair. |
| T-02-INPUT | Tampering | subject_ids / student_id validation | mitigate | `subject_ids.*` `exists:subjects`; `student_id` `exists` scoped to students — malformed/foreign ids rejected before any DB write (V5). |
| T-02-SC | Tampering | package installs | N/A | No packages installed this phase (`02-RESEARCH.md`). |
</threat_model>
