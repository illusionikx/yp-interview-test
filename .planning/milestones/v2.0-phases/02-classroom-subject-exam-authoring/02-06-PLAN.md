## Phase Goal (User Story)

**As a** lecturer, **I want to** build the full teaching content model — classrooms, subjects, class-subject links, student rosters, and complete exams — **so that** everything is authored and correct before any of it is exposed to a student.

*(Derived from the Phase 2 `**Goal:**` line in `.planning/ROADMAP.md`.)*

**This slice:** *As a lecturer, I want to edit and delete questions on a draft exam — but never on a published one — so that exam content stays correct and locked once published.*

<objective>
Complete EXM-05 by delivering question edit/delete and the question-level published-edit gate (D-06, Pitfall 3). Editing a question reuses the Alpine `_form` partial from 02-05 and, for MCQ, replaces the entire option set via delete-and-recreate inside a `DB::transaction` (Pattern 2 — safe because no attempts exist while unpublished). The draft-only gate is enforced in `UpdateQuestionRequest.authorize()` and on the store/destroy paths so a published exam's questions are immutable through every mutating endpoint, not just the exam's own routes.

Together with 02-04's exam-level gate (`ExamPublishTest` + ExamControllerTest published-exam 403s) and 02-05's `StoreQuestionRequest` gate, this closes EXM-05: "edit/delete exam AND its questions only while unpublished." `ExamPublishedEditGateTest` here is the named test from `02-VALIDATION.md` for the question-level half.

Purpose: Publishing must be a real lock — the phase's draft/published contract is only trustworthy if every question mutation is gated.
Output: `ExamQuestionController` edit/update/destroy, `UpdateQuestionRequest`, a question edit view reusing the `_form` partial, per-question edit/delete affordances on `exams/show`, and `ExamPublishedEditGateTest`.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → question edit/update/delete | Untrusted edit payloads (options[], correct_option) cross into `UpdateQuestionRequest` behind Phase-1 `role:lecturer`. |
| published-state boundary | The lock that must hold across every question mutation endpoint, not just the exam's own. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-RBAC | Elevation of Privilege | question edit/update/destroy routes | mitigate | Inside the unchanged Phase-1 `role:lecturer` group; the test asserts a student is 403. |
| T-02-PUB | Tampering | UpdateQuestionRequest.authorize() + store/destroy guards | mitigate | Draft-only gate applied to EVERY mutating question endpoint (store from 02-05, update via authorize(), destroy via abort_if) — Pitfall 3's "gate the questions too, not just the exam"; a forged/replayed edit against a published exam is rejected server-side regardless of hidden UI controls. |
| T-02-MCQ | Tampering | UpdateQuestionRequest.after() + transactional recreate | mitigate | Edit reuses the exactly-one-correct after() hook + min:2 + blank-body rejection; option set is replaced atomically in a transaction with `is_correct` derived server-side from `correct_option` — an edit cannot leave a broken MCQ. |
| T-02-MA | Tampering | question/option update persistence | mitigate | Only type/body/points and server-derived option fields are written; `is_correct`/`exam_id` never taken from raw request input (exam from route binding). |
| T-02-SC | Tampering | package installs | N/A | No packages installed this phase (`02-RESEARCH.md`). |
</threat_model>
