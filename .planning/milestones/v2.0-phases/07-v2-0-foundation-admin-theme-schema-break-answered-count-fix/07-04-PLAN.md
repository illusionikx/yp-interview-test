<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| lecturer request → section management under a subject | Only a lecturer assigned to the subject may create/edit/delete its sections (SEC-03) — route-model binding alone is insufficient |
| lecturer request → subject_user assignment | Assigning/unassigning lecturers must itself be ownership-gated and role-restricted |
| request body → section/enrollment writes | Mass-assignment on the new write paths must be constrained to validated fields |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-10 | Elevation of Privilege (IDOR) | SectionController write actions | mitigate | StoreSection/UpdateSectionRequest authorize() checks `$subject->lecturers()->whereKey(user)->exists()` (per-subject ownership, the SEC-03 divergence from D-09), plus nested-binding `abort_unless($section->subject_id === $subject->id, 404)` in edit/update/destroy |
| T-07-11 | Elevation of Privilege | SubjectLecturerController | mitigate | AssignLecturerRequest authorize() enforces per-subject ownership; user_id restricted to role=lecturer via Rule::exists()->where(); unassign scoped to the bound subject's pivot |
| T-07-12 | Tampering (mass assignment) | Section::create / AssignExamRequest | mitigate | create() writes only validated fields plus server-set subject_id/sequence; AssignExamRequest validates section_ids as distinct integers existing in sections (no arbitrary pivot injection) |
| T-07-13 | Tampering / Availability | orphaned classroom_id roster endpoint | mitigate | ClassroomRosterController/AssignStudentRequest and their routes deleted outright (Pitfall 4) so no endpoint writes to the dropped column |
</threat_model>
