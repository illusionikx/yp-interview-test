# Milestone v3.0: Workflow Restructure & UX Polish

**Status:** ✅ SHIPPED 2026-07-19
**Phases:** 9–14
**Total Plans:** 34 (60/60 requirements)

## Overview

Restructured the portal around one navigation hierarchy (login → dashboard + subject
list → enrollment / class management / class → exam), anchored to a **derived semester
model** (`App\Support\Semester`). Delivered a Flowbite-token design layer under Tailwind 3,
a unified toast + confirm-modal alert system, an **auto-assignment** exam-integrity model
(exam visibility derived from subject enrollment — no manual per-class assignment, no
cross-subject leak), a warn-before-destroy attempt-reset/edit lifecycle, a two-tab lecturer
workspace + tabbed exam editor, a restyled sticky-bar take-exam experience, a site-wide
dark-mode sweep, a wiki-style user manual, an expanded demo seeder covering every status,
and Laravel Dusk browser tests against Herd.

### Phase 9: v3.0 Foundations — Semester Model, Design Tokens, Alerts & Entry Pages

**Depends on**: Nothing (first v3.0 phase; builds on shipped v2.0)

**Success Criteria** (what must be TRUE):

  1. A visitor lands on a branded "Online Examination Portal" page (subtitle *"for Yayasan Peneraju Technical Assessment"*) before logging in, and reaches a login card matching the Flowbite design supplied in `v3.md` — actually styled, confirmed by finding its token classes in the compiled CSS, not by eyeballing a browser-default fallback.

### Phase 10: Exam Integrity — Auto-Assignment & Attempt Lifecycle

**Goal**: Exams reach exactly the right students with no manual assignment step, and a lecturer can revise or reset an exam without silently destroying graded work or stranding a student mid-attempt.
**Depends on**: Phase 9 (semester rule, alert/modal system, `lockAndFinalize()` null-guard)
**Requirements**: INT-02, INT-03, INT-04, CLS-05, CLS-06, CLS-07, EDT-04, FIX-03
**Success Criteria** (what must be TRUE):

  1. A student enrolled in any class of a subject automatically sees and can start every active exam in that subject, with no per-class assignment step — and a student enrolled only in a *different* subject can never see or start it, prevented structurally rather than by list filtering.
  2. A lecturer can toggle an exam between draft and active in both directions, including after students have already attempted it.
  3. Resetting an exam's submissions — or saving an edit to an exam that has been attempted — warns the lecturer first with the concrete count of attempts and graded results at stake, and changes nothing until confirmed.
  4. A student whose attempt was cancelled or reset can take that exam again.
  5. Changing an exam's assignment or status reports its outcome visibly, rather than appearing to return to the same page with nothing changed.

**Plans**: 9/9 plans complete

Plans:

- [x] 10-01-PLAN.md — Wave 0 specs: INT-04 cross-subject negative (all 3 surfaces) + INT-02 five-count correctness
- [x] 10-02-PLAN.md — Wave 0 specs: CLS-07 reset + INT-03 retake, EDT-04 across all 4 mutations, D-5 third crash site
- [x] 10-03-PLAN.md — Disarm the factory trap: pin subject on 16 test fixtures (behavior-neutral, suite stays green)
- [x] 10-04-PLAN.md — `AttemptVoider` (the single voiding authority) + D-5 null-guard + lecturer-safe `AttemptVanishedException`
- [x] 10-05-PLAN.md — Delete the assignment feature (FIX-03 by removal) + CLS-06 bidirectional toggle
- [x] 10-06-PLAN.md — D-1: drop the pivot, derive visibility from subject enrollment, convert ~19 test files
- [x] 10-07-PLAN.md — CLS-07: reset route/action + Submissions panel + warning modal (human-verify)
- [x] 10-08-PLAN.md — EDT-04 backend: relax all 4 gate sites, warn-and-void atomically (D-7), invert 5 gate tests
- [x] 10-09-PLAN.md — EDT-04 frontend: unhide the editor, one shared warning-copy source, 3 forms intercepted (human-verify)

**Notes**: **This is the highest-risk phase in v3.0 and it carries all three deferred design decisions — `/gsd-discuss-phase 10` must resolve them before any plan is written:** (a) *auto-assignment scope* — does CLS-05 drop the `exam_section` pivot and derive visibility from subject enrollment, or keep the pivot and auto-populate it? One decision, two safe paths, one unsafe one: keeping and auto-populating re-opens v2.0's CRITICAL cross-subject leak unless **every** write is scoped by `exam.subject_id === section.subject_id`. (b) *attempt cancel/reset mechanism* — reset-in-place, hard delete, or a schema change to the unique key? It collides with three shipped invariants: `UpdateExamRequest::authorize()` forbids editing a published exam at all, `answers.attempt_id` is `cascadeOnDelete()` (hard delete destroys graded scores), and `attempts.unique(exam_id,user_id)` blocks the very retake INT-03 requires. An earlier "soft-void" guess was retracted for exactly this reason — do not re-guess it. (c) *reset granularity* — per-exam (v3.md's placement) or also per-student?
INT-04 is settled here, in the same phase as CLS-05, by construction. Its acceptance needs an explicit **negative** regression test ("a student in Subject Y's class does NOT see Subject X's exam") — a positive "exam appears" test does not prove the leak stayed closed. CLS-07 and EDT-04 must converge on **one** shared, lock-guarded voiding service (mirroring the `AttemptGrader` precedent), never duplicated across two controllers, and its delete must take the same row lock `Attempt::lockAndFinalize()` takes. Warning copy must differentiate "N students have started but not finished" from "N students have been graded — their scores will be permanently deleted." FIX-03 resolves here because this phase decides its fate: the Phase 9 toaster fixes it if the assignment feature survives, and it disappears entirely if CLS-05 drops the pivot.

### Phase 11: Navigation Restructure — Landing Hierarchy, Dashboard, Subjects & Enrollment

**Goal**: Both roles land on one page after login carrying a dashboard above their subject list, and every destination in the app is reachable by drilling down from there.
**Depends on**: Phase 9 (semester rule, tokens, toasts), Phase 10 (exam visibility model)
**Requirements**: NAV-03, NAV-04, UX-04, DASH-01, DASH-02, DASH-03, DASH-04, SUBJ-01, SUBJ-02, SUBJ-03, SUBJ-04, SUBJ-05, ENR-09, ENR-10, ENR-11
**Success Criteria** (what must be TRUE):

  1. After login, both roles land on a single page showing a gradient welcome banner and role-specific stat cards above their subject list — a lecturer sees classes assigned this and future semesters plus enrolled-vs-seats with a progress bar; a student sees subjects enrolled this semester.
  2. A lecturer sees every subject assigned to them in one ungrouped table, can create/edit/delete subjects, and can open subject & class management from any row.
  3. A student sees their subjects grouped by semester with each lecturer's name, can hide and unhide past semesters, and can open the class page for any subject.
  4. A student can enroll from a single page by choosing subject → class → enroll, in as many subjects as they want, restricted to classes whose enrollment window is currently open.
  5. Every feature a role could reach before the restructure is still reachable by clicking through the new hierarchy, and every back button's label names where it leads.

- [x] 11-02-PLAN.md — Wave 2: lecturer dashboard (3 scoped bounded aggregates) + assigned-subject CRUD hub (DASH-01, DASH-03, SUBJ-01, SUBJ-02)
- [x] 11-03-PLAN.md — Wave 2: student dashboard (2 aggregates) + NEW enrolled-subjects-by-semester list with hide/unhide + enroll button (DASH-04, SUBJ-03, SUBJ-04, SUBJ-05)

**UI hint**: yes

**Notes**: NAV-04 inverts v2.0's "existence ≠ reachability" lesson — this phase *removes* working navbar links, so the acceptance gate runs in both directions: enumerate every destination reachable today (grep `resources/views` for every clickable `route(...)`, not just the navbar) and confirm each has a path in the new tree, rather than only ticking off v3.md's diagram. Results and the help/manual pages are the named at-risk destinations; the manual's new home is the Phase 14 help button, so Phase 11 must not orphan it in the interim. Dashboard "this and future semester" must compare `(year, semester)` as a composite ordinal via Phase 9's helper — a naive `year >= Y AND semester >= S` silently returns wrong results across the S1 rollover. Dashboard cards are bounded aggregates (`COUNT`/`SUM`/`withCount`) only, never a loop over relationships. The student "subject list" (enrolled subjects, grouped by semester) is a genuinely **new** query — it is not the existing `SubjectBrowseController` catalog, which is relabeled as "Class enrollment" (ENR-09) with its logic unchanged. Per Decision #3, `Section` is a UI-copy relabel to "Class" only — do not rename the model, table, FKs, or route names.

### Phase 12: Lecturer Workspace — Class Management, Exam Editor & Grading

**Goal**: A lecturer runs everything about a subject — its classes, its exams, authoring, and grading — from one hub reached from the subject list.
**Depends on**: Phase 10 (integrity model, auto-assignment), Phase 11 (navigation hub)
**Requirements**: CLS-01, CLS-02, CLS-03, CLS-04, CLS-08, EDT-01, EDT-02, EDT-03, EDT-05, GRD-06
**Success Criteria** (what must be TRUE):

  1. Subject & class management opens on a classes tab by default with an exams tab beside it; classes are grouped by semester with hide/unhide for past ones, each row showing its class code and total/max students as a progress bar.
  2. A lecturer can create, edit, and delete a class with max students, location, and enrollment period, and can create, edit, and delete the subject's exams from the exams tab — where each exam shows its grading progress.
  3. A lecturer edits one exam's details and questions on a single page as two tabs, and every exam carries a name.
  4. A lecturer reorders questions with move-up/move-down buttons beside each question's number, reorders each question's answer options, and can shuffle an option arrangement once at authoring time.
  5. The grading page shows the class details, the exam details, grading progress, and the full student list.

### Phase 13: Student Exam Experience — Class Page & Take Exam

**Goal**: A student finds the right exam from their class page and takes it knowing how much time is left, how far along they are, and what the instructions say.
**Depends on**: Phase 10 (exam visibility), Phase 11 (navigation hierarchy)
**Requirements**: TAK-07, TAK-08, TAK-09, TAK-10, TAK-11, TAK-12
**Success Criteria** (what must be TRUE):

  1. The class page shows the subject's detail plus an exam list in its own card, each exam marked not-yet-open / opened / closed and whether it has been taken or graded.
  2. A student can start each exam only once per class — the start button is disabled afterward, blocking a second trip to the exam start page.
  3. The take-exam page keeps a sticky top bar carrying the subject and exam name, the timer, and question progress, with a button that opens the exam instructions in a popup — exam details and instructions read as separate things.
  4. A student navigates questions from a vertical stepper that checkmarks answered questions and still shows those checkmarks after a mid-exam page reload.
  5. At 10 minutes remaining the timer turns red and a toaster appears exactly once; answer options always render in their authored arrangement.

## Milestone Summary

**Key Decisions** (full rationale in the archived REQUIREMENTS + PROJECT.md):

- **#1 August gap** → current semester rolls forward to the upcoming one (never null).
- **#2/#8 Answer arrangement** → authoring-time only; reorder via move-up/down buttons (no drag-and-drop, no runtime randomization).
- **#3 "Class"** → UI-copy relabel of `Section` only; model/table/FKs/routes unchanged.
- **#4 Semester** → derived value object, not a stored table; `ordinal()` gives a total order across the S1 year rollover.
- **#5 Flowbite tokens** → ported token *values* into `tailwind.config.js` (v3 `theme.extend`); did NOT upgrade to Tailwind v4.
- **#6 beforeunload** → NOT automatable (ChromeDriver auto-dismisses); stays a manual check.
- **#7 Dusk DB** → separate `yp-student-exam-dusk` via `.env.dusk.local` + `DatabaseTruncation`; never the demo DB.
- **D-1 Auto-assignment** → dropped the `exam_section` pivot; visibility derived from subject enrollment (closes the cross-subject leak structurally). FIX-03 resolved by removal.

**Defects caught & fixed before close** (each would have broken a shipped promise):

- Phase 11 — a CRITICAL data-loss bug in the navigation restructure (found in code review, fixed with regression test).
- Phase 12 — a CRITICAL question/option reorder bug (found in code review, fixed with regression test).
- Phase 13 — restored an unreachable-result guard (the v2.0 "existence ≠ reachability" lesson).
- Integrity — the third unguarded `lockForUpdate()->first()` on `attempts` (lecturer grading path) null-guarded via `AttemptVanishedException`; T-09-01 now holds repo-wide.
- Milestone-close UI review (phases 9/11/12/13/14, 16–18/24) — x-cloak flash (no CSS rule), a zero-`dark:` page (lecturer/subjects/create), a dropdown missing its dark arm, and an off-brand indigo accent on the take-exam page + question editor. All fixed.

**Technical Debt Incurred:**

- Orphaned `student.subjects.show` (SubjectBrowseController@show + view) — a dead DUPLICATE surface, not a broken flow (enrollment reachable via subjects.index, class page via subjects.class). Safe to remove or wire; left in place to avoid destabilizing the green suite at close.
- Exam-editor option reorder persists on question-save (array-index order) rather than immediately like question reorder — a UX inconsistency, not a data bug. Immediate-persist is a future decision.
- Typography scale undocumented (7 sizes / 3 weights); hierarchical in practice but no documented scale.
