# Roadmap: Online Examination Portal

## Overview

Starting from the existing Laravel 11 + Breeze scaffold (auth already Validated), this roadmap builds the exam-portal domain in dependency order: first the schema and role-gated foundation everything else stands on, then the lecturer-side content model (classrooms, subjects, exams with MCQ/open-text questions), then the assignment layer that makes class-scoped visibility possible, then the correctness-critical attempt-taking flow (server-anchored timer, single-attempt integrity, no answer leakage), then grading and results, and finally a demo seeder and README so the graded deliverable is reproducible from a clean clone. Each phase is a coherent, independently verifiable slice; nothing student-facing is built before the access-control and assignment layers that make it safe to expose.

**v2.0 (Enrollment, Exam Availability & Fixes)** extends this foundation with a schema break: the v1 single-classroom model is replaced by subject-scoped, capacity-and-window-gated sections that students self-enroll into. To minimize gate-count, v2.0 is delivered in exactly two phases. **Phase 7** bundles every foundational workstream that must land before any user-facing enrollment/availability flow can be built: the Flowbite admin shell + dark-mode toggle, the independent answered-count bugfix, and the atomic schema-break slice (section/subject_user/enrollments schema, lecturer section CRUD, subject↔lecturer assignment, the rewritten `Exam::scopeVisibleTo()`, and the seeder/factory rewrite) — all of which must land together so a clean clone never sits in a non-booting state. **Phase 8** builds every remaining student- and lecturer-facing feature on top of that foundation: the enrollment apply/withdraw/capacity/window/rejection flow, the exam availability window with its pre-start page and in-progress safeguards, and finally the non-technical user manuals, written last because they document the shipped v2.0 UI and flows.

## Phase Details

### Phase 1: Foundation — Domain Schema & Role-Based Access Control

**Goal**: All domain tables exist with correct constraints, and two role-gated areas (Lecturer/Student) sit on top of the existing Breeze auth scaffold
**Mode:** mvp
**Depends on**: Nothing (first phase; builds directly on the existing Breeze scaffold)
**Requirements**: RBAC-01, RBAC-02, RBAC-03, RBAC-04
**Success Criteria** (what must be TRUE):

  1. A fresh clone can run migrations against an empty MySQL database, and every domain table (classrooms, subjects, class_subject, exams, questions, options, exam_classroom, attempts, answers) is created with correct foreign keys and the unique constraints needed for single-attempt integrity.
  2. A user who self-registers through the public registration form always ends up with the Student role — there is no public path to create a Lecturer account.
  3. Immediately after logging in, a Lecturer is routed to a lecturer-specific area and a Student is routed to a student-specific area.
  4. A Student who navigates directly to a lecturer-only URL is blocked server-side (403/redirect) by middleware, not merely kept off the navigation menu.

### Phase 4: Attempt-Taking

**Goal**: A student's timed exam attempt is captured reliably and cannot be gamed, duplicated, or lost
**Mode:** mvp
**Depends on**: Phase 3
**Requirements**: TAK-01, TAK-02, TAK-03, TAK-04, TAK-05, TAK-06
**Success Criteria** (what must be TRUE):

  1. Starting an assigned exam creates exactly one timed attempt anchored to a server-recorded start time; a second attempt on the same exam by the same student is rejected at the database level.
  2. The on-screen countdown reflects a server-computed deadline, and any answer submitted after that deadline has passed is rejected server-side regardless of what the client displays.
  3. Answers are saved as they are entered, so refreshing the page or losing the connection mid-exam does not lose previously entered answers.
  4. When the time limit expires, the attempt auto-submits and no further answer changes are accepted.
  5. At no point while taking the exam does the student-facing view, its markup, or its underlying data reveal which option is correct.

### Phase 5: Grading & Results

**Goal**: Every submitted attempt ends in an accurate, appropriately-gated score that both the student and the lecturer can see
**Mode:** mvp
**Depends on**: Phase 4
**Requirements**: GRD-01, GRD-02, GRD-03, GRD-04, GRD-05
**Success Criteria** (what must be TRUE):

  1. Multiple-choice answers are scored automatically the moment an attempt is submitted, with no lecturer action required.
  2. A Lecturer can grade each open-text answer, assigning a score up to that question's point value.
  3. A Student's final result stays hidden until every open-text answer in their attempt has been graded.
  4. Once graded, a Student can view their total score and a per-question breakdown of their own attempt.
  5. A Lecturer can view results per exam and per student across all attempts.

### Phase 8: v2.0 Features — Enrollment, Exam Availability & User Manuals

**Goal**: Students self-enroll into capacity-and-window-gated sections and can withdraw or be rejected with a visible reason, exams carry a start-availability window with a pre-start details page and in-progress safeguards, and non-technical manuals document the finished app end-to-end
**Depends on**: Phase 7
**Requirements**: ENR-01, ENR-02, ENR-03, ENR-04, ENR-05, ENR-06, ENR-07, AVL-01, AVL-02, AVL-03, AVL-04, AVL-05, DEL-04, DEL-05
**Success Criteria** (what must be TRUE):

  1. A student can open a subject's sections page and see live capacity (e.g. "28/30" or "FULL") and an enrollment-window status label for every section, including ones not yet open or already closed, and can apply to an open, non-full section to be enrolled immediately with no approval step; concurrent applies to a nearly-full section never push its enrollment count over capacity.
  2. A student can hold at most one active enrollment per subject per semester, can withdraw from a section before its close date, and — whether previously withdrawn or rejected — can re-apply to a section while its enrollment window is still open; a student cannot apply to a section whose window is not currently open — refused server-side with a clear message, not merely hidden from the UI.
  3. Any lecturer assigned to the subject can reject an enrolled student from a section, choosing a reason from a fixed dropdown, and the rejected student can see that reason.
  4. A lecturer can set an availability window on an exam (optional start and/or end datetime, unbounded when empty); before starting, a student sees a pre-start details page (instructions, duration, availability window, their enrolled section) with Proceed and Back actions, and can only start an attempt within that window (refused otherwise with a clear "not yet open"/"closed" message); once an attempt has started, it runs to completion governed solely by its own timer — a withdrawal, a rejection, or the availability window closing after start never cuts it short — and while an attempt is in progress, trying to close the tab or navigate away triggers a browser confirmation warning that does not appear during an intentional submit or the timer's own auto-submit.
  5. A non-technical student manual (plain language, no setup/CLI content) walks through enrolling in a subject's section, withdrawing, checking exam availability, taking a timed exam, and viewing results; a non-technical lecturer manual walks through managing subjects/sections, viewing rosters and rejecting a student with a reason, authoring/assigning an exam with an availability window, and grading — and both manuals accurately reflect the shipped v2.0 UI and flows.
